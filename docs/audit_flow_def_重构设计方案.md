# audit_flow_def 重构设计方案

## 1. 目标

本方案只讨论设计，不包含数据库变更执行，不包含生产数据修改。

目标是把 `audit_flow_def` 收敛成一张纯粹的“审批流程模板头表”，不再承载实例级语义，也不保留来源不明的自关联兼容字段。

本次设计遵循以下原则：

1. 不改数据库现状，只定义目标模型与字段职责。
2. 流程定义表只负责“模板定义”，不负责“实例运行状态”。
3. 模板头表和步骤定义表职责分离。
4. 一个模板头表字段只表达一个稳定语义。

## 2. 表职责定义

`audit_flow_def` 的唯一职责应为：

- 定义一条审批流程模板
- 给模板提供稳定编码、名称、启用状态和版本信息
- 指明这条模板适用于哪类业务对象

它不应承担以下职责：

- 保存审批实例运行信息
- 保存节点级审批规则细节
- 保存模板自身的自关联兼容字段

## 3. 目标字段模型

### 3.1 保留并作为主流程字段使用

以下字段建议作为主流程白名单字段：

- `id`
  - 流程模板主键

- `flow_code`
  - 模板唯一编码
  - 例如：`PLAN_DISPATCH_STRATEGY`
  - 后端和前端都可用它稳定引用模板

- `flow_name`
  - 模板名称
  - 用于展示和管理后台维护

- `entity_type`
  - 模板适用的业务实体类型
  - 例如：`INDICATOR`、`PLAN_REPORT`
  - 这是模板归属语义的一部分，应保留

- `is_enabled`
  - 模板是否启用
  - 用于控制当前是否可被实例创建逻辑选中

- `version`
  - 模板版本号
  - 用于后续模板演进和兼容控制

- `description`
  - 模板说明
  - 说明这条审批流适用场景和规则概要

- `created_at`
  - 模板创建时间

- `updated_at`
  - 模板更新时间

### 3.2 明确视为重复或不再推荐使用

- `flow_def_id`
  - 当前最像历史兼容或自关联残留字段
  - 目标设计中不再作为主模型字段保留

原因：

- `audit_flow_def` 已经有自己的主键 `id`
- 当前模板头表不需要再额外维护一套“模板定义 ID”
- 若无明确版本链或父模板设计，保留它只会让语义混乱

## 4. 模板适用范围设计

### 4.1 `entity_type` 应保留

虽然数据库备注里提到过“删除了 entity_type 和 biz_type 字段”的历史信息，但当前系统现实上仍然依赖 `entity_type`：

- 后端会按 `entity_type` 查模板
- 审批实例启动时会根据业务对象类型选取可用流程模板

因此目标设计建议：

- 保留 `entity_type`
- 不保留 `biz_type`
- 模板和业务对象的绑定统一通过 `entity_type` 表达

### 4.2 `flow_code` 和 `entity_type` 的关系

建议明确：

- `flow_code` 是模板的唯一业务编码
- `entity_type` 是模板适用范围

也就是说：

- `flow_code` 决定“是哪条模板”
- `entity_type` 决定“适用于哪类业务对象”

两者都需要，但职责不同，不重复。

## 5. 版本设计

### 5.1 `version` 应保留

模板版本号是合理字段，建议保留。

原因：

- 流程模板后续一定会演化
- 即使当前系统尚未正式做“同编码多版本并存”，也应保留版本字段作为未来扩展位

### 5.2 版本字段的职责边界

`version` 只表达：

- 模板定义的版本号

它不表达：

- 当前实例用了哪个节点
- 当前实例运行到了哪里

这些都属于实例层或步骤层语义。

## 6. 启用状态设计

`is_enabled` 应保留。

建议语义：

- `true`
  - 当前模板可被新实例选用

- `false`
  - 当前模板不再用于新实例创建
  - 但旧实例历史仍可追溯

这是模板头表很重要的控制字段，不建议移除。

## 7. 展示与说明字段设计

### 7.1 `flow_name`

`flow_name` 应保留。

它是管理后台和审批配置页的核心展示字段。

### 7.2 `description`

`description` 应保留。

它的职责是：

- 解释模板适用场景
- 说明流程概要
- 帮助运维或配置人员识别模板

## 8. 与 `audit_step_def` 的边界

`audit_flow_def` 负责：

- 模板编码
- 模板名称
- 适用对象类型
- 启用状态
- 版本

`audit_step_def` 负责：

- 步骤顺序
- 步骤名称
- 审批规则
- 审批人解析方式

因此：

- 模板头表不应下沉保存节点细节
- 节点定义表不应反向承担模板头信息

## 9. 目标代码模型建议

如果只从代码设计出发，不改数据库，`AuditFlowDef` 应按以下语义重构：

- 主字段：
  - `id`
  - `flowCode`
  - `flowName`
  - `entityType`
  - `isEnabled`
  - `version`
  - `description`
  - `createdAt`
  - `updatedAt`

- 关联字段：
  - `steps`

- 不再纳入代码主模型：
  - `flowDefId`

## 10. 对当前系统的实际影响

如果未来按此设计推进，代码层需要改动的点主要有：

1. `AuditFlowDef` 实体映射
2. 模板查询逻辑中对 `entity_type` 的使用
3. 模板选择逻辑中对 `flow_code` / `is_enabled` / `version` 的判定
4. 后续 `audit_step_def` 与模板头表的边界梳理

但这些属于后续实现，不属于本设计文件的执行范围。

## 11. 本次设计结论

对 `audit_flow_def` 的目标设计结论如下：

1. `audit_flow_def` 应被定义为纯粹的审批流程模板头表。
2. `flow_code`、`flow_name`、`entity_type`、`is_enabled`、`version` 是主流程白名单字段，应保留。
3. `description` 可保留，承担模板说明语义。
4. `created_at`、`updated_at` 应保留。
5. `flow_def_id` 视为冗余或历史兼容字段，不再纳入目标主模型。
6. `remark` 视为重复说明字段，移出目标主模型。
7. 模板头表不负责节点细节，节点规则统一由 `audit_step_def` 承担。

## 12. 建议的下一步

建议后续按顺序推进：

1. 先确认这份 `audit_flow_def` 目标设计
2. 再设计 `audit_step_def`
3. 最后统一把模板头表、步骤定义表、实例表、节点实例表四层边界收口

这样模板层和实例层才能真正一致。
