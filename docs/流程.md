# 业务流程定义与后端实现分析

> 本文档根据实际业务反馈与架构现状，补充完善了归档证明、流程节点回退、父子指标关系、审批人转办等关键实现细节，确保所有方案均可落地。


## 1. 指标创建与第一层下发 (战略发展部 -> 职能部门)

### 您的描述
*   在任务下创建指标，初始状态为 草稿 (`DRAFT`)。
*   点击“下发”，状态变为 待审批 (`PENDING`)。
*   触发本部门内审批流程。
*   审批驳回 -> 状态退回 草稿 (`DRAFT`)。
*   审批通过 -> 状态变为 已下发 (`DISTRIBUTED`)，指标流向职能部门。


### 后端实现
*   完全符合 `indicator` 表的 `status` 字段设计 (`DRAFT`, `PENDING`, `DISTRIBUTED`)。
*   “下发”操作会创建一个 `audit_instance` 实例，该实例关联一个定义好的“指标下发审批流”（在 `audit_flow_def` 中定义）。
*   审批人通过 `audit_instance` 的 `requester_org_id` 动态确定。
*   审批流的结果（通过/驳回）会通过回调逻辑来更新 `indicator` 表的 `status`。
*   【优化】审批通过后，建议在接口VO中增加“已归档/已生效”状态字段，结合 `audit_step_instance` 审计日志，前端可明确展示“上次归档时间/状态”，为用户提供“已完成”证明。

### 结论
*   完全匹配。后端的通用审批流完美支持了这个环节。

## 2. 指标拆分与第二层下发 (职能部门 -> 学院) 改

### 您的描述
*   职能部门接收到的是“父指标”。
*   职能部门可以将父指标拆分为单个或者多个“子指标”下发给学院。
*   这个下发流程（草稿 -> 待审批 -> 已下发）与第一层使用同一套审批逻辑。


### 后端实现
*   `indicator` 表中的 `parent_indicator_id` 字段仅用于父子指标的关系查找，无需自动进度汇总。
*   业务仅需支持“查找该指标有哪些子指标/父指标”，不做进度聚合。
*   审批流完全复用，`audit_instance` 关联不同实体ID。

### 结论
*   完全匹配。后端的自引用数据结构和可复用流程引擎设计得非常好。

## 3. 学院进度填报与审批 (核心循环)

这是您描述的最核心、最精妙的业务循环，也是最能体现后端设计优势的地方。

### 您的描述
*   学院进行指标填报，填报内容处于 草稿 (`DRAFT`) 状态。
*   点击“提交”，状态变为 待审批 (`PENDING`)。
*   触发本部门（学院）内部的审批流程。
*   审批通过:
    *   审批状态显示“通过”，通过后会提交到对应的职能部门，只能部门那边的审核人再通过才能继续。
    *   填报内容状态退回 草稿 (`DRAFT`)，允许继续编辑。
    *   指标的进度被更新。
*   只要有某个阶段不通过，审批不通过:
    *   审批状态显示“不通过”。
    *   填报内容状态退回 草稿 (`DRAFT`)。
    *   指标的进度不更新。


### 后端实现
*   该流程主体为 `plan_report` 或 `progress_report` 表。
*   学院提交时创建 `audit_instance`。
*   审批通过后：
    1.  更新 `indicator` 表的 `progress` 字段。
    2.  报告状态回到 `DRAFT`，允许下次编辑。
    3.  【优化】建议在 `plan_report` 增加“归档/已生效”字段，或接口聚合“上次归档时间/状态”，便于用户证明历史已完成。
*   审批不通过：报告状态回到 `DRAFT`。

### 结论
*   完全匹配且设计精妙。这个流程不是一个线性的“完成”流程，而是一个周期性的“审核与更新”循环。后端的审批引擎通过不同的回调逻辑，完美地支撑了“通过”和“驳回”两种分支的不同业务效果。

## 4. 多级审批与驳回机制（流程节点回退）

您补充的“多级审批”和“驳回可返回上一节点”是整个审批引擎的核心能力，它并非针对特定业务，而是对所有审批流程通用。


### 后端实现
*   审批流程顺序由 `audit_flow_def`（流程定义表）和 `audit_step_def`（流程步骤表）共同定义。
*   `audit_instance`（流程实例表）记录当前流程走到哪一步，`current_step_id` 指向当前步骤。
*   审批通过：`current_step_id` 顺序推进。
*   驳回：`current_step_id` 回退到上一个节点，通过查询 `audit_step_def` 找到上一节点及其审批人。
*   流程只会一二三四顺序推进，节点可回退但不会“死循环”。
*   所有节点流转、审批人、意见均可通过 `audit_step_instance` 追溯。

### 结论
*   设计完全支持。这个机制形成了一个非常灵活的、可进可退的审批链条，是整个业务流程能够顺畅运转的关键。它将复杂的流程控制逻辑从业务代码中剥离出来，沉淀为通用的平台能力。

## 5. 关键用户体验设计 (UX Principles)

以下三点是确保系统好用、透明、可靠的关键设计，应作为开发和验收的核心准则。

### 1. 审批时间轴 (Approval Timeline)
*   用户需求: 每个指标或报告的填报人，需要能清晰地看到当前审批进行到了哪一步，以及历史的流转路径。例如：`[填报员-提交] -> [副院长-通过] -> [职能部门-审核中]`。
*   实现方案: 后端数据完全支持。
    *   `audit_step_instance` 表记录了每一次审批实例中，每一个步骤的执行人、时间、状态和意见。
    *   前端应基于此数据，为每个审批实例绘制一个清晰、可视化的流程时间轴。

### 2. 强制驳回意见与精准推送
*   用户需求: 审批人驳回时，必须说明理由，且驳回意见能精准推送给上一节点处理人，避免信息断层。
*   实现方案: 后端数据支持，应用层需保障。
    *   数据存储: `audit_step_instance` 表的 `comment` 字段用于存储驳回意见。
    *   强制填写: 应用层（前端+后端）必须校验，当审批操作为“驳回”时，`comment` 字段不可为空。
    *   精准推送: 应用层在完成驳回操作后，必须调用消息服务，根据审批流上下文找到上一节点的处理人，并向其发送包含驳回意见的通知（站内信/手机通知等）。

### 3. 填报版本控制 (Versioning)
*   用户需求: 审批通过后的填报内容应作为历史快照被固化。被驳回后，用户可以修改，但不能丢失历史记录，应有“版本”的概念。
*   实现方案: 通过“数据原地更新 + 完整审计日志”实现。
    *   根据 `plan_report` 和 `plan_report_indicator` 的唯一约束，系统对单次填报活动（如“五月份报告”）的数据采用原地更新策略，即数据库中只保存当前最新版本的数据。
    *   版本追溯的核心由 `audit_step_instance` 表的日志来保障。每一次提交、通过、驳回的操作和意见都被完整记录，形成了一个不可篡改的、详细的“变更日志”，从而实现了事实上的版本追溯和审计能力。

## 6. 审批人不在岗时的异常处理（转办/代办/强制介入）此为后续内容。前期不要求。 

### 需求说明
*   若当前审批人因出差、离职等原因无法处理，流程不能卡死。
*   支持审批任务的“转办”（审批人主动转交他人）、“代办”（管理员或系统指定代理人）、“强制介入”（管理员直接指定新审批人）。

### 表结构建议
*   `audit_instance` 增加 `agent_user_id` 字段，记录当前代理人。
*   `audit_step_instance` 增加 `delegated_by` 字段，记录转办/代办来源。

### 流程实现
*   审批人可在前端发起“转办”，系统将当前节点审批权转交给新用户，记录转办日志。
*   管理员后台可强制指定新审批人，或设置超时自动转交。
*   所有转办/代办操作均写入 `audit_step_instance` 日志，便于追溯。
*   前端应展示“当前审批人已转办/代办，由XX处理”提示。

## 7. 父子指标关系说明（无进度聚合）

*   `indicator` 的 `parent_indicator_id` 仅用于指标之间的来源关系查找。
*   仅支持“查找该指标有哪些子指标/父指标”，不做任何进度合并、聚合、建议值等功能。
*   业务仅需知道某个指标来源于哪个父（负）指标，或有哪些下级指标，除此之外无任何聚合或联动。

【前端/数据看板体验建议】
- 职能部门负责人如需整体掌握下属子指标进度，前端“指标列表页”必须支持一键导出Excel，包含父子关系与各子指标进度。
- 数据看板页面应支持按职能部门、周期等维度聚合展示各项数据，便于管理者全局把控。
- 技术实现：后端提供带父子关系与进度的列表查询接口，前端调用后可直接导出为Excel。

## 8. 填报周期标识与历史归档展示

*   所有填报、审批、归档数据必须强制绑定“周期标识”（如年月、季度等），并在页面、接口VO、审批流日志中显著展示。
*   DRAFT 状态下，页面需同时展示“当前周期”、“上次审批通过的时间与数值”，避免用户混淆。
*   建议接口返回结构中，增加“当前周期”、“上次归档时间/数值”字段，前端大字显示。
*   文档补充：所有填报、审批、归档数据均强制带周期标识，用户可一眼区分本周期与历史数据。

## 9. 数据变更对比与知情权提醒

*   审批流中，若数据被上级修改，系统必须记录变更前后内容，并在下级用户界面以明显标识（如小红点、变更对比弹窗）提醒。
*   建议在 `audit_step_instance` 或专门的“变更日志表”中，记录每次修改的字段、原值、新值、修改人。
*   前端需提供“变更对比”功能，支持用户查看被修改的内容及修改人。
*   文档补充：流程流转中，若数据被上级修改，下级用户界面将收到变更提醒，并可查看详细对比。

【前端UI技术建议】
- 变更对比界面应采用“Git Diff”或“Word修订模式”视觉效果：新增内容高亮绿色，删除内容红底带删除线，细粒度标注具体变动点。
- 技术实现：后端返回变更前后内容，前端用diff算法（如diff-match-patch、jsdiff等）渲染高亮对比，提升可读性和知情权。

## 9.1. 提交撤回功能建议（表与字段详解）后续，只有填报人可以修改填报数据，其他审批人只能查看。

【当前实现】
- 主要涉及表：plan_report（或progress_report）、audit_instance、audit_step_instance。
- 提交后，plan_report.status 字段由 DRAFT 变为 PENDING，audit_instance.status 变为 PENDING。
- 只有审批人驳回（audit_step_instance.status=REJECTED）后，plan_report.status 才能回到 DRAFT。

【建议补充】
- 在不变更表结构的前提下，建议通过业务逻辑实现“撤回”操作：
    1. plan_report.status = PENDING 且 audit_instance.status = PENDING 时，允许发起人点击“撤回”按钮。
    2. 撤回后，plan_report.status 恢复为 DRAFT，audit_instance.status 置为 CANCELLED（或自定义撤回状态，若无此枚举可用CANCELLED代替）。
    3. 在 audit_step_instance 表插入一条新记录，记录 type=WITHDRAW/RECALL，status=CANCELLED，operator=发起人，时间戳等。
    4. 操作日志表（如有）也应记录撤回动作。
    5. 前端在 plan_report.status=PENDING 且当前用户为发起人时展示“撤回”按钮。

## 9.2. 多角色身份与回避原则澄清（表与字段详解）

【当前实现】
- 主要涉及表：sys_user、sys_role、user_role（或类似命名）、audit_instance、audit_step_instance。
- 用户与角色的绑定通过 user_role（user_id, role_id）表实现。
- 审批流分配任务时，audit_step_instance.assignee_id 字段指定审批人。
- 系统设计保证：同一流程（audit_instance）中，user_id 只能作为发起人或某一审批节点的 assignee_id，不会重复。

【回避原则说明】
- 若未来需强制回避，可在 audit_step_def 表增加 avoid_initiator 字段（布尔型），表示该节点审批人不能为发起人。
- 审批流转时，若发现 assignee_id == audit_instance.initiator_id，后端自动查找副职或上级（可通过 user_role 及组织结构表 sys_org、org_user 等实现），并将任务分配给其 user_id。
- 审批日志（audit_step_instance）需记录实际审批人和回避原因。



*   所有代办/转办操作必须在审批时间轴上以特殊UI标识（如“[系统代办]”水印）展示，且审批意见区分“本人/代理人/管理员”。此为后续内容 
*   强制要求代办操作上传授权附件（如截图、公文），并在审批日志中存档。
*   建议接口VO增加“操作类型（本人/代办/转办/系统）”、“附件URL”等字段。
*   文档补充：所有代办/转办操作均需特殊标识，并强制上传授权附件，确保流程严肃性和可追溯性。

【前端交互与合规建议】
- 代办操作提交前，前端弹窗必须有“我已确认获得授权，且知晓本操作将留下不可篡改的系统日志”勾选框，未勾选无法提交。
- 技术实现：前端校验勾选状态，后端日志记录操作人、时间、授权附件等，确保合规可追溯。
