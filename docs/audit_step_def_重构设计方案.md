# audit_step_def 重构设计方案

## 1. 目标

本方案只讨论设计，不包含数据库变更执行，不包含生产数据修改。

目标是把 `audit_step_def` 收敛成一张纯粹的“审批步骤模板定义表”，只负责定义步骤顺序、步骤类型和审批角色规则，不再保留互相重叠的规则字段。

本次设计遵循以下原则：

1. 不改数据库现状，只定义目标模型与字段职责。
2. 步骤定义表只描述模板规则，不保存实例运行结果。
3. 步骤顺序、步骤类型、审批人规则三块语义必须分开。
4. 同一业务语义不允许长期保留两套字段。

## 2. 表职责定义

`audit_step_def` 的唯一职责应为：

- 定义某条审批模板下有哪些步骤
- 定义每个步骤的顺序
- 定义每个步骤的节点类型
- 定义每个步骤由哪个角色负责审批

它不应承担以下职责：

- 保存实例级审批人快照
- 保存步骤运行状态
- 用多套字段重复描述审批角色规则

## 3. 目标字段模型

### 3.1 保留并作为主流程字段使用

以下字段建议作为主流程白名单字段：

- `id`
  - 步骤模板主键

- `flow_id`
  - 所属流程模板 ID
  - 关联 `audit_flow_def.id`

- `step_no`
  - 步骤顺序号
  - 作为唯一正式顺序字段

- `step_name`
  - 步骤名称
  - 用于展示和业务识别

- `step_type`
  - 步骤类型
  - 用于区分“提交节点”“审批节点”等不同步骤语义

- `role_id`
  - 审批角色 ID
  - 关联 `sys_role.id`
  - 表示该步骤应由哪个角色负责审批

- `is_terminal`
  - 是否为终态步骤
  - 用于明确标识流程在哪个节点结束

- `created_at`
  - 步骤模板创建时间

- `updated_at`
  - 步骤模板更新时间

### 3.2 明确视为重复或不再推荐使用

- `approver_type`
  - 当前设计中不再保留
  - 模板层不再负责描述“USER/ROLE/ORG_ROLE”多态解析入口

- `step_code`
  - 如果没有明确的稳定业务编码用途，则和 `step_name + step_no` 组合表达能力重叠
  - 当前目标设计中不作为主模型字段保留

- `approver_id`
  - 当前设计中不再保留
  - 模板层不直接定义具体审批人
  - 具体审批人应在实例启动时根据 `role_id` 和组织上下文动态解析

- `can_skip`
  - 当前无节点跳过功能需求
  - 不纳入目标主模型

- `timeout_hours`
  - 当前没有真正启用超时调度和超时处理逻辑
  - 不纳入目标主模型

## 4. 步骤顺序设计

### 4.1 只保留 `step_no`

当前步骤顺序的正式字段应为：

- `step_no`

原因：

- 它已经承担了步骤排序职责
- `WorkflowApplicationService` 当前也是按 `step_no` 排序
- 步骤实例层的 `step_index` 也应来源于它

目标设计建议：

- `step_no` 作为唯一顺序字段保留
- 不再额外依赖其他字段表达顺序

### 4.2 `step_code` 不参与步骤推进

如果未来确实需要业务编码，可以单独再设计用途。  
但当前设计中，步骤推进和排序都不应依赖 `step_code`。

## 5. 审批角色规则设计

这是 `audit_step_def` 当前最核心的收口点。

### 5.1 主规则

审批角色规则只保留一套：

- `role_id`

它决定“该步骤应由哪个角色负责审批”。

### 5.2 推荐语义

- 模板层定义：
  - `role_id`
  - 表示该节点要求的审批角色

- 实例启动时：
  - 后端根据 `role_id`
  - 再结合 `sys_user_role + sys_user + sys_org`
  - 解析出当前具体审批人

### 5.3 为什么模板层不直接定义具体人

原因：

- 人员会变动
- 同一流程模板要复用于不同组织
- 模板层应该稳定，实例层才落具体审批人

因此目标设计建议：

- 模板层保留 `role_id`
- 具体审批人只在 `audit_step_instance` 落地

## 6. 步骤类型设计

### 6.1 `step_type` 应保留

`step_type` 应作为正式主模型字段保留。

原因：

- 它决定该节点是不是“提交节点”“审批节点”
- 它应该替代现在很多靠 `step_name` 包含“提交”来做判断的逻辑

### 6.2 推荐语义

建议最少收敛为：

- `SUBMIT`
  - 提交节点

- `APPROVAL`
  - 审批节点

如果未来有需要，再扩展：

- `NOTICE`
- `CONFIRM`

但当前不建议先引入太多类型。

### 6.3 不应再靠 `step_name` 猜测步骤类型

当前代码里存在“节点名称包含提交，就当提交节点”的倾向。  
这不应成为长期规则。

目标设计建议：

- `step_type` 决定步骤类型
- `step_name` 只负责展示

## 7. 终态、跳过、超时设计

### 7.1 `is_terminal`

建议保留。

原因：

- 它能明确标识流程在哪个节点结束
- 后续如果出现非最后一步即结束、条件终止、分支终止等情况，这个字段会比单纯依赖顺序更稳
- 这是模板层有价值的控制字段

### 7.2 `can_skip`

当前无需求，建议移出目标主模型。

如果未来真的支持跳过节点，再重新引入。

### 7.3 `timeout_hours`

当前没有真实超时任务和升级机制，不建议保留在主模型中。

否则它会变成“表里有、逻辑不用”的典型空转字段。

## 8. 与其他表的边界

`audit_flow_def` 负责：

- 模板头信息

`audit_step_def` 负责：

- 步骤模板规则

`audit_instance` 负责：

- 某次实例整体状态

`audit_step_instance` 负责：

- 某次实例中某个节点的实际执行结果

因此，`audit_step_def` 不应保存：

- 实际审批人
- 实际审批时间
- 实际审批意见

这些都属于 `audit_step_instance`。

## 9. 目标代码模型建议

如果只从代码设计出发，不改数据库，`AuditStepDef` 应按以下语义重构：

- 主字段：
  - `id`
  - `flowId`
  - `stepNo`
  - `stepName`
  - `stepType`
  - `roleId`
  - `isTerminal`
  - `createdAt`
  - `updatedAt`

- 不再纳入代码主模型：
  - `approverType`
  - `approverId`
  - `stepCode`
  - `canSkip`
  - `timeoutHours`

## 10. 对当前系统的实际影响

如果未来按此设计推进，代码层需要改动的点主要有：

1. `AuditStepDef` 实体映射
2. `WorkflowApplicationService.resolveApproverId(...)`
3. `WorkflowApplicationService.initializeStepInstances(...)`
4. 提交节点识别逻辑
5. 流程模板管理接口和前端配置页

但这些属于后续实现，不属于本设计文件的执行范围。

## 11. 本次设计结论

对 `audit_step_def` 的目标设计结论如下：

1. `audit_step_def` 应被定义为纯粹的审批步骤模板定义表。
2. `flow_id`、`step_no`、`step_name`、`step_type`、`role_id`、`is_terminal` 是主流程白名单字段，应保留。
3. 模板层定义审批角色，不直接定义具体审批人；具体审批人应在实例层动态解析。
4. `approver_type`、`approver_id` 当前移出目标主模型。
5. `step_code` 当前无必要，移出目标主模型。
6. `can_skip`、`timeout_hours` 当前无明确启用需求，移出目标主模型。
7. 步骤类型判断应由 `step_type` 决定，不再依赖 `step_name` 猜测。

## 12. 建议的下一步

建议后续按顺序推进：

1. 先确认这份 `audit_step_def` 目标设计
2. 再回头统一检查四张表的边界是否完全一致：
   - `audit_flow_def`
   - `audit_step_def`
   - `audit_instance`
   - `audit_step_instance`
3. 最后再决定是否进入代码模型调整

这样模板层和实例层的职责才会真正闭合。
