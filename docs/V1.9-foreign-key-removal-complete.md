## 💡 为什么删除外键约束

### 1. 分布式数据库的限制
- ❌ **跨库关联困难**: 外键无法跨数据库实例引用
- ❌ **性能问题**: 外键验证需要跨节点网络调用,增加延迟
- ❌ **扩展性差**: 数据分片(sharding)时外键成为瓶颈
- ❌ **维护复杂**: 多个数据库实例间外键状态难以同步

### 2. 应用层约束的优势
- ✅ **更灵活**: 业务逻辑可动态调整,无需DDL操作
- ✅ **性能更好**: 避免外键验证开销和锁等待
- ✅ **易于测试**: 可模拟数据验证场景
- ✅ **支持最终一致性**: 适配微服务和分布式架构
- ✅ **避免死锁**: 减少数据库层面的锁竞争

### 3. 实际案例
```java
// ❌ 数据库外键约束(已删除)
// ALTER TABLE indicator ADD CONSTRAINT fk_indicator_owner_org
//   FOREIGN KEY (owner_org_id) REFERENCES sys_org(id);

// ✅ 应用层验证(推荐)
@Service
public class IndicatorService {

    @Transactional
    public void createIndicator(Indicator indicator) {
        // 验证组织ID存在性
        if (!orgService.exists(indicator.getOwnerOrgId())) {
            throw new OrgNotFoundException("组织不存在: " + indicator.getOwnerOrgId());
        }

        // 验证任务ID存在性
        if (!taskService.exists(indicator.getTaskId())) {
            throw new TaskNotFoundException("任务不存在: " + indicator.getTaskId());
        }

        // 保存指标
        indicatorRepository.save(indicator);
    }
}
```
