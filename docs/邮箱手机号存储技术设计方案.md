# SISM 邮箱与手机号存储技术设计方案

更新时间：2026-04-18

## 1. 文档目标

本文用于明确 SISM 中邮箱、手机号这类联系方式数据的技术落地方案，解决当前前后端字段存在但后端未真正持久化、接口返回与数据库模型不一致、以及安全边界不清晰的问题。

本文聚焦以下范围：
- 管理员在用户管理中维护邮箱、手机号
- 用户本人在个人中心维护邮箱、手机号
- 联系方式的数据库落库、接口返回、权限控制与日志审计

本文**不包含**以下内容：
- 短信发送能力
- 邮件发送能力
- 手机验证码 / 邮箱验证码完整验证流程
- 外部通讯录同步平台的正式接入

## 2. 当前现状

### 2.1 前端现状

当前前端已经把邮箱、手机号当作正式字段使用：
- `UserManagement.vue` 的创建/编辑用户弹窗包含 `email`、`phone` 输入项
- 用户列表表格包含邮箱展示列
- `BasicInfo.vue` 个人中心页面包含 `email`、`phone` 编辑项
- 前端 `CreateUserInput` / `UpdateUserInput` 类型也都定义了 `email`、`phone`

### 2.2 后端现状

当前后端存在明显模型漂移：
- `AuthController.CreateUserRequest` / `UpdateUserRequest` 暴露了 `email` 字段，但 `phone` 尚未进入请求 DTO
- `AuthController.UserListItemResponse` 虽然定义了 `email`、`phone`，实际组装时仍返回 `null`
- `UserService.createUser()` / `updateUser()` 方法签名虽然带 `email` 参数，但当前未真正写入任何实体字段
- `UserProfileController` 当前只更新 `realName` 和 `avatar`，并未处理 `email`、`phone`
- `User` 实体没有 `email`、`phone` 字段

### 2.3 数据库现状

现有 `V1__baseline_current_schema.sql` 中，`sys_user` 表仅包含：
- `id`
- `username`
- `password_hash`
- `real_name`
- `sso_id`
- `org_id`
- `is_active`
- `created_at`
- `updated_at`

即：**当前基线表结构并未存储邮箱和手机号**。

另外，当前仓库还存在一个需要在后续实现时同步处理的现状风险：
- `User` 实体与种子数据中已出现 `avatar_url`、`token_version`
- 但基线迁移与实体定义并非完全一致

这说明 IAM 相关模型存在一定的“代码 / 迁移 / 种子数据”漂移，后续新增联系方式能力时必须严格以 Flyway 迁移为准统一模型。

### 2.4 测试与模拟数据现状

当前仓库中的真实库表并未存储邮箱、手机号；但前端 mock 数据中存在示例邮箱和手机号。这些 mock 值仅用于前端演示，不应视为正式生产数据来源。

当前 seed 账号数据主要是测试账号矩阵，不包含真实联系方式字段，说明系统目前尚未正式采集和托管真实联系人信息。

## 3. 设计目标

本次设计目标如下：
- 让邮箱、手机号具备正式持久化能力
- 让管理端创建用户、编辑用户、查询用户时能读写联系方式
- 让个人中心能维护本人联系方式
- 将联系方式与认证核心字段解耦，避免继续向 `sys_user` 主表堆字段
- 为后续“联系方式校验、通知发送、SSO 同步、数据治理”预留扩展点
- 明确安全策略，避免真实联系方式在接口、日志、审计和备份链路中无控制扩散

## 4. 设计原则

### 4.1 账户主表与资料扩展解耦

`sys_user` 继续承载认证账户主信息：
- username
- password_hash
- real_name
- org_id
- is_active
- token_version

邮箱、手机号属于**个人资料 / 联系方式扩展信息**，不建议继续堆进 `sys_user` 主表。

### 4.2 优先保持现有接口语义稳定

管理端和个人中心前端都已经把邮箱、手机号视为用户资料的一部分，因此后端改造优先保持调用方式稳定，只补足持久化与权限逻辑，不大改前端交互模型。

### 4.3 安全优先，但不过度设计

一期采用：
- 独立表隔离
- 严格鉴权
- 接口 no-store
- 审计与日志脱敏
- 最小权限数据库访问

二期如有更高合规要求，再引入字段级加密或 KMS 托管密钥。

## 5. 总体方案

### 5.1 方案结论

**采用独立的 `sys_user_profile` 表存储邮箱和手机号，不直接加到 `sys_user`。**

### 5.2 选择该方案的原因

- **职责清晰**：`sys_user` 负责认证，`sys_user_profile` 负责资料扩展
- **改造风险更小**：不会影响现有登录、鉴权、角色等主链路
- **扩展性更好**：后续可继续承接头像、联系来源、验证状态、个性化设置等字段
- **安全边界更清晰**：联系方式单独建模后，更易做权限、脱敏和审计隔离
- **与已有个人中心设计文档方向一致**：只是本次将范围收敛到邮箱 / 手机号能力优先落地

## 6. 数据库设计

### 6.1 新增表：`sys_user_profile`

建议采用一对一扩展表，主键直接使用 `user_id`，避免无意义代理主键。

建议字段如下：

| 字段 | 类型 | 可空 | 说明 |
| --- | --- | --- | --- |
| `user_id` | `BIGINT` | 否 | 主键，同时外键关联 `sys_user(id)` |
| `email` | `VARCHAR(255)` | 是 | 原始邮箱展示值 |
| `email_normalized` | `VARCHAR(255)` | 是 | 标准化邮箱，用于唯一校验 |
| `phone` | `VARCHAR(32)` | 是 | 原始手机号展示值 |
| `phone_normalized` | `VARCHAR(32)` | 是 | 标准化手机号，用于唯一校验 |
| `email_verified` | `BOOLEAN` | 否 | 邮箱是否已验证，默认 `false` |
| `phone_verified` | `BOOLEAN` | 否 | 手机号是否已验证，默认 `false` |
| `contact_source` | `VARCHAR(32)` | 否 | 联系方式来源 |
| `source_remark` | `VARCHAR(255)` | 是 | 来源补充说明 |
| `created_at` | `TIMESTAMP` | 否 | 创建时间 |
| `updated_at` | `TIMESTAMP` | 否 | 更新时间 |

### 6.2 联系方式来源枚举

`contact_source` 一期使用字符串枚举，建议值：
- `ADMIN_CREATE`
- `ADMIN_UPDATE`
- `SELF_UPDATE`
- `BATCH_IMPORT`
- `SSO_SYNC`
- `UNKNOWN`

这样后续可以回答“这条联系方式从哪里来”的治理问题。

### 6.3 标准化规则

#### 邮箱标准化
- 去首尾空格
- 全部转小写
- 空字符串按 `null` 处理

#### 手机号标准化
- 去首尾空格
- 去除中间空格、短横线
- 一期默认支持中国大陆手机号，保存时建议归一化为 11 位纯数字
- 空字符串按 `null` 处理

### 6.4 唯一性约束

建议通过标准化字段建立**部分唯一索引**：
- `email_normalized` 非空唯一
- `phone_normalized` 非空唯一

这样可以避免：
- `Admin@xx.com` 与 `admin@xx.com` 重复
- `138 0013 8000` 与 `13800138000` 重复

### 6.5 建议 DDL

```sql
CREATE TABLE public.sys_user_profile (
    user_id BIGINT PRIMARY KEY,
    email VARCHAR(255),
    email_normalized VARCHAR(255),
    phone VARCHAR(32),
    phone_normalized VARCHAR(32),
    email_verified BOOLEAN NOT NULL DEFAULT FALSE,
    phone_verified BOOLEAN NOT NULL DEFAULT FALSE,
    contact_source VARCHAR(32) NOT NULL DEFAULT 'UNKNOWN',
    source_remark VARCHAR(255),
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT fk_sys_user_profile_user
        FOREIGN KEY (user_id) REFERENCES public.sys_user(id)
        ON DELETE CASCADE
);

CREATE UNIQUE INDEX uk_sys_user_profile_email_normalized
    ON public.sys_user_profile (email_normalized)
    WHERE email_normalized IS NOT NULL;

CREATE UNIQUE INDEX uk_sys_user_profile_phone_normalized
    ON public.sys_user_profile (phone_normalized)
    WHERE phone_normalized IS NOT NULL;
```

### 6.6 为什么不直接修改 `sys_user`

不选“直接在 `sys_user` 增加 `email` / `phone` 字段”的原因：
- 认证主表字段职责会继续膨胀
- 后续若再增加验证状态、来源、同步信息，会导致主表继续膨胀
- 不利于权限隔离和按需查询
- 不利于未来对敏感资料做独立审计和治理

## 7. 后端改造设计

### 7.1 领域与仓储

在 `sism-iam` 中新增：
- `domain/UserProfile.java`
- `domain/repository/UserProfileRepository.java`
- `infrastructure/persistence/JpaUserProfileRepository.java`
- `infrastructure/persistence/JpaUserProfileRepositoryInternal.java`

`UserProfile` 建议只承载资料扩展字段，不与认证逻辑耦合。

### 7.2 应用服务

#### `UserService`

需要调整为真正持久化联系方式：
- `createUser(...)`：创建 `User` 后同步创建 `UserProfile`
- `updateUser(...)`：更新联系方式时对 `UserProfile` 执行 upsert
- 增加邮箱、手机号标准化与唯一性校验

#### `UserProfileService`

需要扩展当前个人中心逻辑：
- `getProfile` 时合并读取 `User + UserProfile`
- `updateProfile` 时支持 `email`、`phone`
- 更新本人联系方式时写入 `contact_source = SELF_UPDATE`

### 7.3 控制器改造

#### `AuthController`

需要调整：
- `CreateUserRequest` 新增 `phone`
- `UpdateUserRequest` 新增 `phone`
- 管理员创建 / 编辑用户时将 `email`、`phone` 真正传入服务层并落库
- `UserListItemResponse` 组装邮箱、手机号时从 `UserProfile` 读取

#### `UserProfileController`

需要调整：
- `UserProfileResponse` 增加 `email`、`phone`
- `UpdateProfileRequest` 增加 `email`、`phone`
- 当前 `avatar` 与联系方式更新逻辑拆分清晰，避免继续形成“头像 / 姓名 / 联系方式”混合更新的隐式逻辑

### 7.4 查询性能设计

管理端用户列表是分页接口，不能对每一条用户记录单独查一次资料表。

建议实现方式：
- 先分页查 `User`
- 收集当前页 `userIds`
- 批量查询 `sys_user_profile`
- 组装成 `Map<Long, UserProfile>` 后回填列表响应

这样避免 N+1 查询。

如果后续列表字段继续增多，也可以改为 repository 层直接做分页 join 投影。

## 8. 接口设计

### 8.1 管理员创建用户

接口保持：
- `POST /api/v1/auth/users`

请求体新增 / 明确字段：

```json
{
  "username": "zhangsan",
  "password": "Admin123",
  "realName": "张三",
  "email": "zhangsan@example.com",
  "phone": "13800138000",
  "orgId": 44,
  "roles": ["REPORTER"]
}
```

规则：
- `email`、`phone` 都为可选
- 如填写则必须通过格式校验
- 若与已有标准化值冲突，返回 400 业务错误

### 8.2 管理员更新用户

接口保持：
- `PUT /api/v1/auth/users/{id}`

请求体新增 / 明确字段：

```json
{
  "realName": "张三",
  "email": "zhangsan@example.com",
  "phone": "13800138000",
  "orgId": 44,
  "roles": ["REPORTER", "APPROVER"]
}
```

### 8.3 管理员用户列表

接口保持：
- `GET /api/v1/auth/users`

返回结构继续保留：
- `email`
- `phone`

说明：
- 一期为了兼容现有前端编辑流，管理员列表接口继续返回**完整联系方式**
- 该能力仅对现有管理员组织访问开放
- 若后续要进一步收缩暴露面，可新增“详情接口”并在列表场景改成脱敏显示

### 8.4 个人中心查询

接口保持：
- `GET /api/v1/profile`

响应新增：

```json
{
  "id": 124,
  "username": "admin",
  "realName": "系统管理员",
  "orgId": 35,
  "orgName": "战略发展部",
  "email": "admin@example.com",
  "phone": "13800138000",
  "avatar": null,
  "roles": ["系统管理员"]
}
```

### 8.5 个人中心更新

接口保持：
- `PUT /api/v1/profile`

请求体建议调整为：

```json
{
  "realName": "系统管理员",
  "email": "admin@example.com",
  "phone": "13800138000"
}
```

说明：
- 联系方式更新与头像上传建议分离
- `avatar` 继续走独立接口更清晰

## 9. 权限与安全设计

### 9.1 访问权限

#### 管理员侧
- `/api/v1/auth/users` 继续沿用当前管理员组织访问控制
- 只有具备管理员组织访问资格的用户才能查看、创建、编辑其他用户联系方式

#### 用户本人侧
- `/api/v1/profile` 只允许当前登录用户访问自己的联系方式
- 不允许普通用户通过该接口查看他人联系方式

### 9.2 日志安全

禁止在应用日志中直接打印完整联系方式：
- 不记录完整邮箱
- 不记录完整手机号
- 异常日志只记录 `userId`、操作类型、校验错误类型

建议新增脱敏工具：
- 邮箱：`zh***@example.com`
- 手机：`138****8000`

### 9.3 审计策略

如后续接入统一审计，建议记录：
- 谁修改了联系方式
- 修改的是哪个用户
- 修改类型是邮箱还是手机号
- 修改来源是管理员还是本人

但**不建议在审计表中直接保存完整旧值 / 新值**。

推荐审计文案：
- `用户 124 更新了邮箱`
- `管理员 124 更新了用户 223 的手机号`

### 9.4 缓存与浏览器存储

当前系统已对 `/api/` 响应统一加 `Cache-Control: no-store` 等响应头，这一点继续保留，可降低浏览器缓存泄露敏感信息的风险。

前端侧应避免：
- 将邮箱、手机号长期缓存到本地持久化存储
- 将完整联系方式打到调试日志中

### 9.5 数据库访问权限

数据库层面建议：
- 应用账号保留正常读写权限
- 非应用类运维 / 分析账号默认不直接读取 `sys_user_profile`
- 如确需查询，应走审批或只读受控账号
- 备份数据同样视作敏感数据，继承同等访问控制要求

### 9.6 二期加密预留

如学校后续将该字段纳入更高等级个人信息保护要求，二期可升级为：
- 数据库存密文列（如 `email_ciphertext` / `phone_ciphertext`）
- 保留标准化索引列用于唯一性校验
- 密钥由 KMS 或环境变量托管

本次一期**先不做字段级加密**，以控制实现复杂度，优先完成正式入库和权限治理。

## 10. 数据来源与采集设计

### 10.1 一期允许的数据来源

一期仅允许以下方式写入：
- 管理员在用户管理页手工录入
- 用户本人在个人中心手工维护
- 经审批的批量导入任务

### 10.2 一期不纳入的来源

一期不直接接入：
- 教务 / 人事外部系统自动同步
- 邮件系统回写
- 短信平台回写
- 第三方通讯录自动拉取

### 10.3 存量数据策略

当前系统历史库中没有正式联系方式数据，因此迁移时：
- 不做联系方式回填
- 所有现有用户默认无 `sys_user_profile` 数据或为空资料记录
- 由后续管理员维护或用户本人首次补录

### 10.4 测试数据策略

- `seed` 数据中继续不写真实联系方式
- 示例邮箱统一使用 `example.com`、`sism.local` 等虚拟域名
- 示例手机号统一使用测试号段或脱敏值

## 11. 迁移与发布方案

### 11.1 Flyway 迁移

建议新增迁移脚本：
- `V72__create_sys_user_profile.sql`

脚本职责：
- 创建 `sys_user_profile` 表
- 创建唯一索引
- 补充必要注释

### 11.2 发布顺序

建议按以下顺序上线：
1. 先发数据库迁移
2. 再发后端读写能力
3. 再发前端表单与列表联调版本
4. 最后补测试数据与文档

### 11.3 兼容策略

在后端尚未完全改完前：
- 查询无资料记录的用户时，`email` / `phone` 返回 `null`
- 前端继续按空值展示 `-`
- 避免因为资料表为空导致旧用户列表报错

## 12. 测试设计

### 12.1 后端测试

建议补以下测试：
- 创建用户时能保存邮箱、手机号
- 编辑用户时能更新邮箱、手机号
- 邮箱重复时返回明确错误
- 手机号重复时返回明确错误
- 个人中心更新本人联系方式成功
- 非管理员无法查看他人联系方式
- 无资料记录用户查询时返回 `null` 而不是报错

### 12.2 前端联调验证

建议验证：
- 管理员创建用户后重新加载列表能看到联系方式
- 管理员编辑联系方式后列表与弹窗回显一致
- 用户本人修改个人中心联系方式后再次进入仍能回显
- 空值显示为 `-`
- 非法邮箱 / 非法手机号前端拦截，后端再次兜底校验

## 13. 风险与待确认项

### 13.1 当前代码模型漂移风险

在实现前，需要先统一以下事实来源：
- `sys_user` 当前正式字段集合
- `avatar_url` 是否已经通过正式迁移落库
- `token_version` 的当前正式版本边界

否则后续 Flyway 编号和实体映射容易再次产生偏差。

### 13.2 手机号校验范围

当前前端偏向中国大陆手机号校验；若学校后续存在国际号码场景，需要在一期结束前决定：
- 是继续限制 11 位大陆手机号
- 还是扩展为国际号码格式

### 13.3 列表是否需要脱敏

当前方案为了兼容管理端编辑体验，管理员用户列表继续返回完整联系方式。

如果后续认为列表级暴露面过大，可在二期改为：
- 列表返回脱敏值
- 编辑弹窗单独调用详情接口获取完整联系方式

## 14. 最终结论

本次建议采用**独立 `sys_user_profile` 表**承接邮箱与手机号，避免继续污染 `sys_user` 认证主表。

一期优先实现：
- 正式数据库落库
- 管理端 / 个人中心双入口写入
- 严格鉴权
