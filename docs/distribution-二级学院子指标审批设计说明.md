# `/distribution` 二级学院子指标下发与审批设计说明

## 1. 文档目标

本文档用于明确 `http://localhost:3500/distribution` 页面所承载的业务边界、状态来源、前后端职责和后续改造原则。

目标只有一个：

- 在**不改动**“战略发展部 <-> 职能部门”既有 `plan` 下发审批链路的前提下，
- 为“职能部门 <-> 二级学院”补齐一条**独立的子指标下发/上报审批链路**，
- 并确保 `/distribution` 只负责职能部门侧的下发与审批，学院填报与发起上报统一收敛到 `/indicators`。

需要特别强调的是：

- “职能部门 <-> 二级学院”虽然是单独的一条业务链路
- 但其**下发、撤回、上报审批、审批历史**的处理模式，应整体参考“战略发展部 -> 职能部门”现有流程
- 本次设计的重点不是重新发明一套流程，而是在二级学院场景中复用同一套流程思想和交互边界

---

## 2. 业务边界

系统当前存在两条不同层级的业务链路：

### 2.1 战略发展部 <-> 职能部门

- 业务对象：`plan`
- 场景：战略发展部向职能部门下发，职能部门向战略发展部上报
- 当前状态：该链路已经完整，不作为本次设计的改造对象

### 2.2 职能部门 <-> 二级学院

- 业务对象：子指标集合
- 场景：职能部门向二级学院下发子指标，二级学院在 `/indicators` 页面填报并发起上报，职能部门审批
- 容器对象：`plan`
- 页面入口：
  - 职能部门侧：`/distribution`
  - 学院侧：`/indicators`
- 当前状态：页面已具备部分能力，但状态来源混乱，仍夹杂上层 `plan` 链路逻辑

### 2.3 本次设计结论

`/distribution` 页面必须只处理 **“职能部门 <-> 二级学院”** 这条链路中的“职能部门侧动作”。

以下能力**不应**再作为 `/distribution` 的主逻辑：

- 战略发展部 -> 职能部门的 `plan` 提交审批
- `plan` 级撤回审批
- 使用总 `plan.status` 直接锁定二级学院子指标流程

---

## 3. 核心设计原则

### 原则 1：不破坏既有 `plan` 链路

“战略发展部 <-> 职能部门”现有链路已完整，必须原样保留：

- 保留既有 `plan` 相关 API
- 保留既有 workflow 定义
- 保留既有页面和状态判断

本次只新增或收敛 `/distribution` 对应的二级链路。

这里需要进一步明确：

- “战略发展部 <-> 职能部门”通过 `plan` 承载一批任务/指标
- “职能部门 <-> 二级学院”也统一通过 `plan` 承载一批子指标
- 也就是说，二级学院链路不是脱离 `plan` 的临时拼装流程，而是使用 `plan.plan_level = FUNC_TO_COLLEGE` 作为正式容器

### 原则 2：`/distribution` 页面只认子指标链路

`/distribution` 内展示、按钮、审批抽屉、撤回动作，均应以“当前职能部门下发给当前学院的子指标集合”为业务主体，只处理：

- 下发前编辑
- 发起下发审批
- 查看下发审批进度
- 撤回已下发但仍可撤回的批次
- 查看并处理学院上报后的审批

这里的“处理方式”应与“战略发展部 -> 职能部门”保持一致：

- 下发是按当前页面选中对象的一整批内容统一发起
- 上报审批也是按当前学院这一批内容统一进入审批
- 不以单条指标作为独立审批主体

### 原则 3：审批进度以审批表/工作流表为准

“职能部门 -> 二级学院”这一层的审批进度，必须以审批表中的真实审批状态为准，而不是以前端补字段为准。

真实来源包括：

- `audit_instance`
- `audit_step_instance`
- workflow detail API

兼容来源包括：

- `plan_report`
- `plan_report_indicator`

其中：

- `plan_report` / `plan_report_indicator` 反映的是“学院在 /indicators 中的填报/上报数据”
- `audit_instance` / `audit_step_instance` 反映的是“审批结果与审批进度”

**最终审批态应以审批表真值为准。**

同时保留以下投影约定：

- 下发流程 `id = 2` 成功走到最后一个节点后，`plan.status` 映射为“已下发”
- 学院上报流程 `id = 4` 的审批结果，不直接覆盖下发容器 `plan.status`
- 学院上报的业务结果主要落在 `plan_report` 及对应审批历史上

### 原则 4：页面只提供批量撤回，不提供单条撤回

业务约束已经明确：

- `/distribution` 页面不提供单条撤回
- 只允许按“当前职能部门 + 当前学院 + 当前子指标集合”执行批量撤回
- 批量撤回规则对齐“战略发展部 -> 职能部门”既有逻辑，只允许在 `pending` 状态撤回

---

## 4. 当前页面存在的问题

## 4.1 上层 `plan` 逻辑污染了 `/distribution`

当前页面仍在读取当前部门的 `plan.status` 并用它控制：

- 是否允许编辑
- 审批按钮文案
- 是否锁定页面

这会导致“二级学院子指标链路”被错误绑定到“上层总 plan 链路”。

### 结果

- 上层 `plan` 进入 `PENDING` 后，二级学院子指标页面被整体锁死
- 页面职责从“子指标链路工作台”变成了“子指标 + plan 混合工作台”

这与业务边界不符。

## 4.2 当前页面的状态来源不统一

当前 `/distribution` 页存在 3 类状态来源并存：

### A. 指标表状态

- `indicator.status`

### B. 兼容字段/前端补字段

- `progressApprovalStatus`
- `canWithdraw`
- `statusAudit`

### C. 审批表/工作流状态

- `audit_instance.status`
- `audit_step_instance.status`
- workflow detail API 返回的审批任务状态

### 结果

- 列表页、按钮显隐、审批抽屉可能出现不同步
- 有些状态刷新后回跳
- 有些状态只是前端 store 本地 patch，并非数据库真实值

## 4.3 当前页面把“审批态”和“内容态”混在一起

当前实现里，单条子指标常被当成独立流程主体来判断：

- 草稿
- 已下发
- 待审批
- 已通过

但从业务上看，`/distribution` 实际处理的是：

- 某个职能部门
- 对某个学院
- 一批子指标

因此页面头部状态和批量动作才是主流程；单条子指标更多是这批次中的内容项。

## 4.4 当前页面曾混入单条撤回语义

业务已经确认：

- 只保留批量撤回
- 单条撤回不是正式业务能力

因此 `/distribution` 中任何单条撤回语义都应视为历史残留，不应继续扩展。

---

## 5. 正确的业务主体

`/distribution` 页面中的真正业务主体不是总 `plan`，也不是完全独立的单条指标，而是：

**当前职能部门对某一所二级学院下发的子指标集合。**

因此页面核心维度应为：

- 当前年度
- 当前职能部门
- 当前选中学院
- 当前学院下的子指标集合

围绕这个主体，页面可以提供：

- 新增子指标
- 编辑草稿子指标
- 批量下发
- 查看学院上报进度
- 进入审批中心处理学院上报审批
- 批量撤回

这里的“批量下发 / 批量审批”都不是对某一条指标逐条处理，而是：

- 对当前二级学院下的全部指标统一发起下发
- 对当前二级学院当前批次的上报统一进入审批

这个“当前批次”的载体，统一定义为：

- 一条 `FUNC_TO_COLLEGE` 的 `plan`
- 该 `plan` 下当前学院对应的一组 `indicator`
- 学院围绕这条 `plan` 产生多次 `plan_report`

---

## 6. 状态来源设计

## 6.1 内容数据来源

子指标内容、权重、里程碑、进度等结构性数据，来自业务表：

- `indicator`
- `milestone`
- `plan_report_indicator`

## 6.2 审批进度来源

审批进度、当前节点、是否待审批、是否已驳回、是否已撤回，来自审批链路：

- `audit_instance`
- `audit_step_instance`
- `/api/v1/workflows/instances/...`

## 6.3 页面判断优先级

`/distribution` 页面中，审批态判断建议遵循以下优先级：

1. 优先使用 workflow detail / 审批表真实状态
2. 若审批表无实例，再回退到业务表中的上报状态
3. 前端兼容字段仅作临时兜底，不作为最终真值

## 6.4 当前链路对应的流程定义

结合当前种子数据，`/distribution` 页面涉及的“职能部门 -> 二级学院”下发动作，当前应明确映射到：

- `audit_flow_def.id = 2`
- `flow_code = PLAN_DISPATCH_FUNCDEPT`
- `flow_name = Plan下发审批（职能部门）`

对应步骤定义来自 `audit_step_def.flow_id = 2`：

1. `填报人提交`
2. `职能部门审批人审批`
3. `分管校领导审批`

这意味着当前文档语境下，“二级学院子指标下发”在工作流模板层面，复用的是 **职能部门下发流程**，而不是 `audit_flow_def.id = 4`。

## 6.5 与 `PLAN_APPROVAL_COLLEGE` 的边界

当前种子中还存在：

- `audit_flow_def.id = 4`
- `flow_code = PLAN_APPROVAL_COLLEGE`
- `flow_name = Plan审批流程（二级学院）`

你已经确认当前这条链路实际分为两段流程：

- 职能部门下发流程使用 `PLAN_DISPATCH_FUNCDEPT (id = 2)`
- 二级学院上报审批流程使用 `PLAN_APPROVAL_COLLEGE (id = 4)`

因此本文档在当前阶段应明确：

- `/distribution` 的批量下发动作用 `PLAN_DISPATCH_FUNCDEPT`
- `/indicators` 是学院填报并发起 `PLAN_APPROVAL_COLLEGE` 的页面
- `/distribution` 只负责查看并处理 `PLAN_APPROVAL_COLLEGE` 产生的审批任务
- `id = 4` 不是下发流程，而是学院从 `/indicators` 发起的正式上报审批流程
- 两段流程的节点推进、撤回边界、审批历史展示方式，均参考“战略发展部 -> 职能部门”现有流程
- 两段流程都围绕同一条 `FUNC_TO_COLLEGE plan` 展开，而不是绕开 `plan` 单独建容器
- 下发流程通过后，`plan.status` 明确映射为数据库既有“已下发”状态

---

## 7. 页面能力边界

## 7.1 本页面应保留的能力

- 按学院查看子指标
- 新增子指标
- 编辑草稿子指标
- 批量下发给当前学院
- 查看当前学院的下发审批进度
- 查看当前学院的上报审批进度
- 进入审批中心处理学院上报审批
- 批量撤回当前学院已下发子指标

## 7.2 本页面不应承载的能力

- 学院在 `/indicators` 中的指标填报
- 学院发起上报
- `plan` 级提交审批
- `plan` 级撤回审批
- 战略发展部 -> 职能部门的总下发链路
- 单条子指标撤回

---

## 8. 前端设计要求

## 8.1 页面主状态

页面头部状态不应直接复用总 `plan.status`，而应改为“当前学院子指标集合”的业务状态。

状态判断依据应来自：

- 该学院下子指标是否存在草稿
- 是否存在已发起的下发审批实例
- 是否存在学院在 `/indicators` 发起的上报审批实例
- workflow 当前审批状态

## 8.2 审批数量

“待审批数量”不应只依赖前端 `progressApprovalStatus` 兼容字段。

应优先来自：

- 当前学院对应业务实体的 workflow pending task 数量

## 8.3 审批抽屉 / 审批中心

学院上报审批的前端交互，已存在统一 UI 组件“审批中心”，`/distribution` 不再单独发明审批交互，而应与该组件对齐。

审批中心或审批详情展示时，应展示：

- 当前学院子指标集合对应的审批流程
- 当前流程状态
- 当前节点
- 审批历史

其中需要明确区分两类实例：

- 下发阶段：`PLAN_DISPATCH_FUNCDEPT`
- 学院上报审批阶段：`PLAN_APPROVAL_COLLEGE`

若继续复用 `ApprovalProgressDrawer` 或其他现有展示能力，其数据实体也必须明确落到这两类实例之一，而不是错误复用上层 `plan`。

## 8.4 按钮规则

建议保持以下规则：

- 存在草稿子指标：显示“新增指标”“下发”
- 下发审批进行中：显示“查看审批”“批量撤回”
- 已下发且学院尚未上报：显示“查看审批”
- 学院已在 `/indicators` 发起上报：显示“查看审批”“进入审批中心”
- 不显示单条撤回

按钮规则中的“批量撤回”必须继续收紧为：

- 只有当前批次处于 `pending` 时才允许撤回
- 一旦下发流程已过可撤回节点，或学院上报审批已经进入处理阶段，则不允许撤回
- 该规则直接参考“战略发展部 -> 职能部门”的下发撤回逻辑，不额外创造新口径

---

## 9. 后端设计要求

## 9.1 不修改现有 `plan` 审批链路

后端现有：

- `PlanApplicationService`
- `PlanController`
- `plan` workflow

继续服务上层链路，不作为本次二级链路的主入口。

## 9.2 为二级链路提供独立业务识别

后端需要能够识别：

- 当前职能部门
- 当前学院
- 当前子指标集合
- 当前审批实例

当前阶段不强制新增独立业务实体模型，先对齐现有任务维度，至少保证能够用 `task_id`、责任部门、目标学院定位到当前批次和对应实例，避免继续和总 `plan` 混用。

这里进一步收敛为：

- 二级学院链路正式绑定 `plan`
- 下发链路围绕 `plan.id` + 当前学院展开
- 学院上报链路围绕 `plan_report.id` 展开
- 不新增新表，不调整现有数据库结构

## 9.3 批量撤回必须限制业务范围

批量撤回不能只按：

- `ownerOrgId`
- `targetOrgId`

粗粒度查询。

必须限制在当前业务范围内，例如：

- 当前计划下的任务集合
- 当前年度
- 当前学院对应的子指标集合

否则会误撤历史数据。

并且撤回状态限制必须对齐既有链路：

- 仅 `pending` 可撤回
- 不能按 `distributed / approved / rejected` 这类前端历史兼容状态粗暴判断
- 应以当前批次对应 workflow instance 的真实节点状态作为准入条件

“撤回后恢复原样”在当前链路中，具体应理解为：

- 当前下发流程实例被取消/撤回，不再继续向后推进
- `plan.status` 回退到下发前对应的可编辑状态，而不是保留“已下发”
- 当前学院这批指标重新回到可编辑、可再次发起下发的状态
- 不删除既有审批痕迹，撤回动作仍应保留在审批历史或审计记录中
- 学院侧尚未完成的新一轮上报流程不应继续向后流转

## 9.4 后端流程选择要求

后端在处理这条链路对应 workflow instance 时，应根据业务动作显式选择流程模板，不能靠页面名称或业务描述模糊匹配。

建议至少满足：

- 执行“下发”时，创建 `flow_def_id = 2`，并绑定当前 `FUNC_TO_COLLEGE plan`
- 学院在 `/indicators` 执行“上报”时，先基于当前 `plan` 生成一条新的 `plan_report`，再创建 `flow_def_id = 4`
- `/distribution` 处理学院审批时，只查询和审批 `flow_def_id = 4` 对应任务
- 详情查询时能反查出当前实例实际使用的 `flow_code`
- 前端展示审批中心或审批抽屉时，以实例真实 `flow_def_id` 为准

并且需要增加一条明确的状态同步规则：

- `PLAN_DISPATCH_FUNCDEPT` 最后一个节点审批通过后，同步将对应 `plan.status` 更新为“已下发”
- 若 `PLAN_DISPATCH_FUNCDEPT` 未通过，则 `plan.status` 不进入“已下发”
- `PLAN_APPROVAL_COLLEGE` 的通过/驳回，不回写成新的下发状态，而是体现在对应 `plan_report.status` 与审批历史中

也就是说，这条链路不是“一个页面默认只映射一个流程模板”，而是：

- `/distribution` 下发动作 -> `PLAN_DISPATCH_FUNCDEPT`
- `/indicators` 学院上报动作 -> 基于 `plan_report` 发起 `PLAN_APPROVAL_COLLEGE`
- `/distribution` 审批动作 -> 处理 `PLAN_APPROVAL_COLLEGE` 的审批任务

需要强调：

- 这里处理的是“当前学院当前批次”的统一审批
- 不是单条指标自身审批
- 当前页面不应再保留“某一个指标 submit/approve/reject 就代表整个学院链路审批”的理解

---

## 10. 数据模型收敛建议

当前前端仍残留以下兼容字段：

- `progressApprovalStatus`
- `canWithdraw`
- `statusAudit`

这些字段的历史作用主要是：

- 前端本地推导状态
- 在真实审批链路不完整时充当过渡字段

但在当前设计下，应逐步收敛为：

- 审批结果：来自审批表/工作流
- 填报数据：来自 `plan_report` / `plan_report_indicator`
- 结构化业务数据：来自 `indicator` / `milestone`

前端兼容字段可以继续保留一段时间用于平滑过渡，但不能继续作为主判定来源。

## 10.1 当前数据库表之间的业务关系

当前数据库已经具备支撑这条链路的基础表，不需要为本次业务改表。应按如下关系理解：

1. `plan`
   - 表示一批同时下发的内容集合
   - 职能部门给二级学院下发的这一批子指标，落在 `plan.plan_level = FUNC_TO_COLLEGE`
   - `plan.created_by_org_id` 是当前职能部门
   - `plan.target_org_id` 是当前二级学院
   - 对于“职能部门 <-> 二级学院”链路，`plan` 就是正式业务容器，不再额外新增新容器表

2. `indicator`
   - 表示该批次中的具体子指标内容
   - `indicator.task_id` 绑定上游任务
   - `indicator.owner_org_id` 表示当前职能部门
   - `indicator.target_org_id` 表示当前二级学院
   - `indicator.status` 只是内容侧投影，不作为审批真值

3. `plan_report`
   - 表示二级学院对当前 `plan` 的一次上报记录
   - 学院在 `/indicators` 页面每发起一次上报，就新生成一条 `plan_report`
   - 不限制同一个月只能提交一次，也不限制同一条 `plan` 只能提交一次
   - 也就是说，学院可以多次上报，每次上报都对应一条新的 `plan_report`
   - 每一条新的 `plan_report`，都必须一一对应一条新的学院上报审批流
   - 因此“学院上报审批”并不是孤立动作，而是有 `plan_report` 作为业务承载对象

4. `plan_report_indicator`
   - 表示学院针对每个指标的具体填报内容
   - 与 `plan_report` 形成主从关系
   - 与 `indicator` 通过 `indicator_id` 建立关联
   - 学院发起上报前，必须保证当前 `plan` 下要求上报的全部指标，都已有对应的 `plan_report_indicator`
   - 如果当前 `plan` 下仍有指标没有对应填报记录，则不允许发起上报审批

5. `audit_instance`
   - 表示某个业务对象的一次审批实例
   - 下发流程实例使用 `PLAN_DISPATCH_FUNCDEPT`
   - 学院上报审批实例使用 `PLAN_APPROVAL_COLLEGE`
   - 它是审批真值主表
   - 对于学院多次上报场景，每次新的 `plan_report` 都应启动一条新的 `audit_instance`
   - 因此可以直接收敛为：`一条 plan_report = 一条审批流`

6. `audit_step_instance`
   - 表示某个审批实例当前推进到哪个节点、由谁在点击时完成了审批
   - 这里要特别说明：系统平时按角色控制谁可以审批，不预先固定具体审批人
   - 只有实际有人点击“通过 / 驳回”时，才记录这个操作人是谁

7. `audit_flow_def` + `audit_step_def`
   - 定义流程模板和节点顺序
   - `id = 2` 定义职能部门下发流程
   - `id = 4` 定义二级学院上报审批流程

因此这条链路的关系可以概括为：

- 职能部门先按 `plan + indicator` 组织当前学院的一批子指标
- 下发时创建 `PLAN_DISPATCH_FUNCDEPT` 对应的 `audit_instance`
- 学院填报时写入 `plan_report + plan_report_indicator`
- 学院每次发起上报时，都新建一条 `plan_report`
- 每条新的 `plan_report` 都单独创建一条 `PLAN_APPROVAL_COLLEGE` 对应的 `audit_instance`
- 最终审批状态一律以 `audit_instance + audit_step_instance` 为准

## 10.3 不改表结构前提下的字段语义收敛

在不修改数据库表结构的前提下，建议将当前字段语义统一为：

1. `plan`
   - 只承担“这批下发内容的业务容器”职责
   - `status` 对应下发流程整体状态投影
   - 当 `id = 2` 下发流程审批通过后，映射为“已下发”

2. `plan_report`
   - 只承担“学院某次上报”的业务容器职责
   - 一次上报对应一条记录
   - 多次上报就多条记录
   - 每条记录单独对应一条 `PLAN_APPROVAL_COLLEGE` 审批流

3. `plan_report_indicator`
   - 只承担“本次上报里每条指标的填报内容”职责
   - 发起上报前要先补齐

4. `indicator.status`
   - 只作为内容投影或兼容展示字段
   - 不再承担审批主真值职责

5. `audit_instance.entity_id`
   - 下发流程可绑定 `plan.id`
   - 学院上报审批流程可绑定 `plan_report.id`

这样处理后，不需要改表，也能把“plan 容器”“多次上报”“每次新流程”三件事同时落地。

## 10.2 审批人与角色的关系

这一点必须和“战略发展部 -> 职能部门”保持一致：

- 流程模板节点只定义“审批角色”
- 页面和权限控制只判断“当前用户是否属于该节点允许审批的角色范围”
- 未发生审批动作前，不需要预先写死具体审批人
- 只有当实际用户点击通过/驳回时，才在节点实例中记录该操作人

因此，本次设计不需要为了“预先指定审批人”去改数据库结构。

## 12. 当前状态流转口径

截至当前实现，`/distribution` 的业务状态统一收敛为：

1. 草稿
2. 下发审批中
3. 已下发待学院填报
4. 学院已上报
5. 审批中
6. 已通过
7. 已驳回
8. 已撤回

其中：

- “下发”对应发起 `PLAN_DISPATCH_FUNCDEPT`
- “学院上报”由 `/indicators` 基于当前 `plan` 新建一条 `plan_report` 后发起 `PLAN_APPROVAL_COLLEGE`
- `/distribution` 中的“学院已上报 / 审批中 / 已通过 / 已驳回” 需要绑定具体 `PLAN_APPROVAL_COLLEGE` 实例真值
- “已撤回” 只允许批量撤回，不保留单条撤回语义

补充说明：

- 页面展示上可以出现“待审批 / 已撤回 / 已通过 / 已驳回”等审批结果标签
- 但它们本质上都应来自审批实例和节点实例的真实推进结果
- 一旦审批走完最后一个节点，该批次就进入审批历史，由现有前端审批历史 UI 承接展示
- 因此不需要再额外发明一套独立于审批历史之外的终态模型
- `plan.status` 在本层只关心“下发是否已完成”，不承载学院每次上报的审批终态

## 13. 已确认的实现口径

基于你刚确认的业务口径，当前阶段统一收敛为以下规则：

1. “下发”固定使用 `id = 2`；“学院上报审批”固定使用 `id = 4`。
2. 学院在 `/indicators` 页面填报并发起上报；`/distribution` 不是学院发起申报的页面。
3. `/distribution` 中处理学院上报审批时，前端直接对齐现有“审批中心”组件，不再额外设计新审批入口。
4. 批量撤回沿用战略发展部 -> 职能部门既有下发逻辑，只有 `pending` 状态允许撤回。
5. “当前学院子指标集合”不再单独发明新口径，而是对齐现有任务维度；学院指标来源于上游“战略发展部 -> 职能部门”已筛选出的任务集合，本层只需继续绑定 `task_id`、当前责任部门、当前下发目标部门 / 学院。
6. 同一学院同一年允许多次下发，并沿用既有下发逻辑保留每次下发记录；后续 workflow detail 查询必须具备按批次或按实例区分的能力，不能只按“学院 + 年度”粗粒度聚合。
7. 当前页面处理的是“某一个二级学院的全部指标统一下发、统一进入上报审批”，不是指标自身逐条审批。
8. 当前数据库字段与审批逻辑已能支撑该业务，本次不修改数据库结构，只明确表之间的职责和关联关系。
9. “职能部门 <-> 二级学院”这一层也正式绑定 `plan` 作为容器，使用 `plan.plan_level = FUNC_TO_COLLEGE`。
10. 学院发起上报前，必须强制要求当前 `plan` 下要求填报的指标都已有对应 `plan_report_indicator`，否则禁止提交。
11. 学院可以多次上报；每次上报都新建一条 `plan_report`，并启动一条新的 `PLAN_APPROVAL_COLLEGE` 审批流程。
12. 下发流程 `id = 2` 成功通过后，`plan.status` 明确映射为数据库既有“已下发”状态。
13. 固定采用“每一条 `plan_report` 对应一条审批流”的实现方式，作为后端建流、前端查流、测试验收的统一口径。

## 13.1 实施检查点

为确保实施过程可检查，建议按以下清单验收：

1. 创建 `FUNC_TO_COLLEGE plan` 后，`/distribution` 能按 `plan + 学院` 正确加载当前子指标集合。
2. 发起下发时，系统创建一条 `PLAN_DISPATCH_FUNCDEPT` 审批实例，并能从 `plan.id` 反查。
3. `id = 2` 最后一个节点通过后，`plan.status` 更新为“已下发”。
4. 学院在 `/indicators` 发起一次上报时，系统先创建一条新的 `plan_report`。
5. 若当前 `plan` 下存在未填报指标，没有完整的 `plan_report_indicator`，则本次上报被拒绝。
6. 每次成功发起上报后，都能看到一条新的 `PLAN_APPROVAL_COLLEGE` 审批实例，且其 `entity_id` 对应本次 `plan_report.id`。
7. 职能部门在审批中心处理的是 `plan_report` 对应的审批流，而不是单条 `indicator` 审批。
8. 同一学院连续上报两次后，应存在两条 `plan_report`，并对应两条独立审批流，互不覆盖。

## 14. 本文档的最终结论

`/distribution` 页面不是上层 `plan` 审批页的一个变体，而是：

**职能部门与二级学院之间，围绕子指标集合展开的独立下发与审批工作台。**

因此：

- 不能继续混用上层 `plan` 状态
- 不能继续依赖前端补状态作为主真值
- 审批进度必须以审批表/工作流状态为准
- 学院填报与发起上报统一在 `/indicators`
- `/distribution` 只承担职能部门侧的下发、查看进度、进入审批中心处理审批
- 既有“战略发展部 <-> 职能部门”链路必须保持不动

后续所有改造均应以此为基准。
