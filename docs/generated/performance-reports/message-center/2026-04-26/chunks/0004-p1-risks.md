## 04 P1 级放大项与结构风险

### 1. 工作流读模型虽有分页，但上层把分页抵消掉了

`WorkflowReadModelService.getMyPendingTasks()` 自身是分页查询，但消息中心的聚合逻辑会从第 1 页开始一直翻到最后一页，把所有待办全部拉完。结果是：

- 底层虽然支持分页，但上层又重新变成全量化
- 用户待办越多，跨模块调用次数越多
- 待办链路会把消息中心的访问成本继续放大

### 2. enrich 阶段存在轻度 N+1 放大

工作流待办组装过程中还会补充计划、组织、用户、流程定义等附加信息。虽然服务内部做了请求级缓存，但缓存只在单次调用内有效；一旦消息中心不断全量翻页，这些 enrich 查询仍会持续叠加。

### 3. 去重位置偏后

当前通常是先把工作流待办和通知都取出来，再基于 `approvalInstanceId` 或实体 identity key 在应用层去重。这样会带来三种浪费：

- 重复候选数据先被取回
- 重复对象先被创建
- 排序与过滤在丢弃前已经付出过成本

### 4. 索引需要结合真实 SQL 再核验

这次没有做数据库 `EXPLAIN`，但从访问模式看，后续应重点核验：

- 通知表的 `recipient_user_id + created_at`
- 通知表的 `recipient_user_id + status + created_at`
- 审批步骤表的 `approver_id + status`
- 审批步骤表的 `instance_id`

这些索引是否合理，会直接决定分页下推后的实际收益上限。
