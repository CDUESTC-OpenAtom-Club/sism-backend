## 05 优化方案

### 方案 A：把分页、过滤、统计下推到查询层

这是收益最高的一步，建议把消息中心拆成两条路径：

- **摘要路径**：只统计待审批数、审批结果未读数、提醒未读数、系统未读数
- **列表路径**：按 `category`、`keyword`、`readState`、`includeRisk`、`createdAt` 直接查当前页

目标是把复杂度从“和用户总消息量强相关”改成“和当前页大小强相关”。

### 方案 B：为 workflow 详情增加精准查询

新增按 `instanceId + taskId + userId` 的单条读取能力，避免详情接口复用全量待办扫描逻辑。这样既能降低详情尾延迟，也能让接口语义更清晰。

### 方案 C：中期演进为分源分页归并或统一读模型

- **短期路线**：workflow 和 notification 各自分页后，在应用层做小范围 merge-sort
- **长期路线**：建立消息中心专属读模型，让所有消息以统一结构沉淀后再分页查询

如果追求快速落地，建议先走短期路线；如果追求长期维护性与扩展性，建议最终落到统一读模型。

### 方案 D：补充观测能力

建议同时增加以下指标，避免优化只停留在代码层：

- 消息中心各接口平均耗时与 P95
- 工作流待办翻页次数
- 单次请求聚合候选数
- 列表接口当前页命中数
- 慢 SQL 与索引命中情况
