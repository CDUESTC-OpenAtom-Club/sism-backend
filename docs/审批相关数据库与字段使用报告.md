# 审批相关数据库与字段使用报告

> 生成时间：2026-03-18
> 数据来源：
> - 线上 PostgreSQL：`strategic@175.24.139.148:8386`
> - 后端代码：`sism-workflow`、`sism-shared-kernel`
> - 当前正式审批主链路：`AuditFlowDef -> AuditStepDef -> AuditInstance -> AuditStepInstance`

## 1. 结论摘要

当前系统中，和审批/工作流直接相关的数据库表一共查到 7 张：

1. `audit_flow_def`
2. `audit_step_def`
3. `audit_instance`
4. `audit_step_instance`
5. `audit_log`
6. `workflow_task`
7. `workflow_task_history`

但按“线上真实审批是否在使用”来分，不能把这 7 张表等量看待。

### 1.1 目前真正承载线上审批流程的核心表

1. `audit_flow_def`
2. `audit_step_def`
3. `audit_instance`
4. `audit_step_instance`

这 4 张表是当前线上审批真正跑起来的主干。

### 1.2 当前存在、但没有承载线上审批主链路数据的辅助/历史表

1. `audit_log`
2. `workflow_task`
3. `workflow_task_history`

它们在代码里有模型，也有表，但当前线上审批主链路没有实际把审批状态落到这些表里：

- `audit_log`：当前线上记录数为 `0`
- `workflow_task`：当前线上记录数为 `0`
- `workflow_task_history`：当前线上记录数为 `0`

所以如果你问“我们有没有使用所有审批相关表”，答案是：

- 有定义并保留这些表
- 但当前线上审批流程真正使用的是前 4 张核心表
- 后 3 张表目前更像保留能力、历史设计或辅助扩展，并未进入当前主流程

## 2. 当前审批主流程的数据流

当前审批从数据库视角看，真实数据流如下：

1. 在 `audit_flow_def` 中按 `flow_code` 找到流程模板
2. 在 `audit_step_def` 中取出该流程的步骤定义
3. 发起审批时，在 `audit_instance` 中创建一条审批实例主记录
4. 同时在 `audit_step_instance` 中为该实例展开多条步骤实例
5. 审批过程中，主要更新：
   - `audit_instance.status`
   - `audit_instance.current_step_index`
   - `audit_instance.completed_at`
   - `audit_instance.result`
   - `audit_step_instance.status`
   - `audit_step_instance.approver_id`
   - `audit_step_instance.comment`
   - `audit_step_instance.approved_at`

### 2.1 当前线上真实流程定义样例

当前线上库里已经存在的流程定义包括：

- `PLAN_DISPATCH_STRATEGY`：Plan 下发审批（战略发展部）
- `PLAN_DISPATCH_FUNCDEPT`：Plan 下发审批（职能部门）
- `PLAN_REPORT_FUNC`：职能部门月度填报审批
- `PLAN_REPORT_COLLEGE`：二级学院月度填报审批
- `INDICATOR_DEFAULT_APPROVAL`：指标审批流程（职能部门）
- `INDICATOR_COLLEGE_APPROVAL`：指标审批流程（二级学院）
- `REPORT_APPROVAL`：报告审批

其中这次线上重点排查的就是 `PLAN_DISPATCH_STRATEGY`。

## 3.1 `audit_flow_def`

### 3.1.1 表作用

审批流定义表。用于保存“一个业务类型应该走什么审批模板”。

### 3.1.2 当前记录数

`7`

### 3.1.3 代码中的使用位置

- `WorkflowApplicationService.getAuditFlowDefByCode(...)`
- `WorkflowApplicationService.resolveAndAttachFlow(...)`
- `JpaWorkflowRepository.findAuditFlowDefByCode(...)`
- `JpaWorkflowRepository.findAuditFlowDefsByEntityType(...)`

### 3.1.4 字段说明

| 字段 | 含义 | 当前用途 | 是否主流程使用 |
|---|---|---|---|
| `id` | 流程定义主键 | 被 `audit_instance.flow_def_id`、步骤定义表引用 | 是 |
| `flow_code` | 流程编码 | 代码按编码选流程，如 `PLAN_DISPATCH_STRATEGY` | 是 |
| `flow_name` | 流程名称 | 展示用，也用于标题/管理 | 是 |
| `is_enabled` | 是否启用 | 代码会过滤只取启用流程 | 是 |
| `created_at` | 创建时间 | 管理和审计 | 次要 |
| `updated_at` | 更新时间 | 管理和审计 | 次要 |
| `description` | 描述 | 辅助说明 | 次要 |
| `version` | 版本 | 管理用途，当前代码未做复杂版本切换 | 次要 |
| `remark` | 备注 | 当前代码未直接使用 | 否，偏保留字段 |
| `flow_def_id` | 旧/冗余字段 | 当前代码未映射 | 否，疑似历史遗留 |
| `entity_type` | 适用业务类型 | 代码用它筛选流程，如 `INDICATOR`、`PLAN_REPORT` | 是 |

### 3.1.5 关键使用方式

#### 发起审批时

后端先看实例上是否已有 `flowDefId`：

- 有：直接使用
- 没有：按 `flow_code` 优先查
- 还没有：再按 `entity_type` 查启用流程

#### 本次实际例子

`TASK` 类型会被代码标准化为 `INDICATOR`，然后优先命中：

- `PLAN_DISPATCH_STRATEGY`
- `PLAN_DISPATCH_FUNCDEPT`
- `INDICATOR_DEFAULT_APPROVAL`

## 3.2 `audit_step_def`

### 3.2.1 表作用

审批步骤定义表。保存一个流程里每一步该叫什么、按什么顺序、由谁审批。

### 3.2.2 当前记录数

`15`

### 3.2.3 代码中的使用位置

- `AuditFlowDef.steps`
- `WorkflowApplicationService.initializeStepInstances(...)`
- `WorkflowApplicationService.resolveApproverId(...)`

### 3.2.4 字段说明

| 字段 | 含义 | 当前用途 | 是否主流程使用 |
|---|---|---|---|
| `id` | 步骤定义主键 | 被步骤实例引用、用于排序兜底 | 是 |
| `flow_id` | 所属流程定义 ID | 将步骤归属到 `audit_flow_def` | 是 |
| `step_name` | 步骤名称 | 如“填报人提交”“战略发展部负责人审批” | 是 |
| `step_type` | 步骤类型 | 区分 `SUBMIT` 与 `APPROVAL`，作为节点语义主字段 | 是 |
| `role_id` | 审批角色 ID | 作为审批人解析规则的主字段 | 是 |
| `can_skip` | 是否允许跳过 | 当前代码未真正执行跳过逻辑 | 暂未真正使用 |
| `timeout_hours` | 超时小时数 | 当前未看到超时调度逻辑使用 | 暂未真正使用 |
| `step_code` | 步骤编码 | 当前主实体未映射 | 否，疑似后续扩展 |
| `is_terminal` | 是否终态步骤 | 当前主实体未映射 | 否 |
| `created_at` | 创建时间 | 管理用途 | 次要 |
| `updated_at` | 更新时间 | 管理用途 | 次要 |
| `step_no` | 步骤顺序 | 代码映射为 `stepOrder`，用于排序和生成 `stepIndex` | 是 |

### 3.2.5 关键使用方式

#### 步骤展开

发起实例时，代码会读取步骤定义，按如下规则排序：

1. `step_no`
2. 若 `step_no` 为空，再按 `id` 兜底

然后逐个生成 `audit_step_instance`。

#### 审批人解析

目标设计口径如下：

- 通过 `role_id` 解析审批角色
- 通过 `userRepository.findByRoleId(...)` 找角色下的用户
- 优先选和发起组织 `requester_org_id` 相同的活跃用户
- 没有同组织用户时，取最小 ID 的候选用户
- `SUBMIT` 节点直接回退为发起人本人

#### 本次修复涉及点

`PLAN_DISPATCH_STRATEGY` 的第一步“填报人提交”属于 `SUBMIT` 节点，因此会被回退成发起人本人。现在代码在展开步骤后，会自动把这种提交节点标记为已完成，再把下一步真正审批节点切成 `PENDING`。

### 3.3.4 字段说明

| 字段 | 含义 | 当前用途 | 是否主流程使用 |
|---|---|---|---|
| `id` | 实例主键 | 主键，步骤实例外键指向它 | 是 |
| `status` | 实例状态 | `IN_REVIEW` / `APPROVED` / `REJECTED` / `WITHDRAWN` | 是 |
| `started_at` | 发起时间 | 发起实例时写入 | 是 |
| `created_by` | 历史字段 | 当前主实体未映射 | 否，疑似旧字段 |
| `created_at` | 历史/扩展时间列 | 当前主实体未映射 | 否 |
| `updated_at` | 历史/扩展时间列 | 当前主实体未映射 | 否 |
| `completed_at` | 完成时间 | 整个流程完成或拒绝时写入 | 是 |
| `current_step_index` | 当前步骤序号 | 用于定位当前待审批步骤 | 是 |
| `entity_id` | 业务实体 ID | 例如任务 ID、报告 ID | 是 |
| `entity_type` | 业务实体类型 | 如 `TASK`、`PlanReport`、`PLAN_REPORT` | 是 |
| `flow_def_id` | 所用流程定义 ID | 指向 `audit_flow_def.id` | 是 |
| `is_deleted` | 逻辑删除标记 | Hibernate `@Where(is_deleted = false)` 生效 | 是 |
| `requester_id` | 发起人用户 ID | 查询“我发起的审批”、选择同组织审批人 | 是 |
| `requester_org_id` | 发起人组织 ID | 角色审批人选择同组织优先 | 是 |
| `result` | 审批结果摘要 | 用于保存“Approved step xxx by user ...”等 | 是 |
| `title` | 实例标题 | 页面展示和业务识别 | 是 |
| `biz_id` | 业务扩展字段 | 当前主实体未映射 | 否 |
| `current_step_id` | 历史字段 | 当前主实体未映射 | 否 |
| `finished_at` | 历史字段 | 与 `completed_at` 语义重叠 | 否 |
| `instance_id` | 历史字段 | 当前主实体未映射 | 否 |
| `flow_id` | 历史字段 | 与 `flow_def_id` 语义重叠 | 否 |

### 3.3.5 关键使用方式

#### 发起审批

发起时主要写入这些字段：

- `flow_def_id`
- `entity_type`
- `entity_id`
- `status = IN_REVIEW`
- `requester_id`
- `requester_org_id`
- `current_step_index`
- `started_at`

#### 查询我的待办

不是直接靠 `audit_instance.requester_id` 查，而是：

- 查 `audit_instance.status = IN_REVIEW`
- 连接 `audit_step_instance`
- 找 `audit_step_instance.status = PENDING`
- 并且 `audit_step_instance.approver_id = 当前用户`

#### 审批通过/拒绝

审批时主要更新：

- 当前步骤状态
- `current_step_index`
- `result`
- 如果已无下一步，再写：
  - `status = APPROVED` 或 `REJECTED`
  - `completed_at`

## 3.4 `audit_step_instance`

### 3.4.1 表作用

审批步骤实例表。它是“谁当前该审批什么”的真正来源。

### 3.4.2 当前记录数

`69`

### 3.4.3 代码中的使用位置

- `WorkflowApplicationService.initializeStepInstances(...)`
- `AuditInstance.approve(...)`
- `AuditInstance.reject(...)`
- `JpaWorkflowRepository.findPendingAuditInstancesByUserId(...)`

### 3.4.4 字段说明

| 字段 | 含义 | 当前用途 | 是否主流程使用 |
|---|---|---|---|
| `id` | 步骤实例主键 | 每一步的主键 | 是 |
| `instance_id` | 所属审批实例 ID | 指向 `audit_instance.id` | 是 |
| `step_name` | 步骤名称 | 展示用，也用于“提交类第一步”判断 | 是 |
| `approved_at` | 审批完成时间 | 通过/拒绝时记录 | 是 |
| `approver_id` | 当前步骤审批人 ID | 待办查询的关键字段 | 是 |
| `comment` | 审批意见 | 通过/拒绝时记录，也用于系统自动完成说明 | 是 |
| `status` | 步骤状态 | `WAITING` / `PENDING` / `APPROVED` / `REJECTED` | 是 |
| `step_index` | 步骤顺序号 | 用于节点排序和推进 | 是 |
| `step_def_id` | 关联步骤定义 | 节点实例与模板步骤的正式关联键 | 是 |
| `approver_org_id` | 审批人组织 ID | 组织路由结果字段 | 设计保留 |
| `created_at` | 创建时间 | 生成步骤实例时写入 | 是 |

### 3.4.5 关键使用方式

#### 初始化步骤实例

发起审批时，系统会为每个步骤定义生成一条步骤实例：

- 第 1 步：`status = PENDING`
- 后续步骤：`status = WAITING`
- 同时写入：
  - `step_index`
  - `step_name`
  - `approver_id`
  - `created_at`

#### 本次修复的核心写法

如果第一步满足：

- `status = PENDING`
- `step_name` 包含“提交”
- `approver_id = requester_id`

则自动更新为：

- 第一步 `status = APPROVED`
- 第一步 `approved_at = now`
- 第一步 `comment = 系统自动完成提交流程节点`
- 第二步从 `WAITING` 变成 `PENDING`

#### 查询待办的关键字段

页面“我的待审批”查的不是 `audit_instance.current_step_index` 本身，而是：

- `audit_step_instance.approver_id = 当前用户`
- `audit_step_instance.status = PENDING`

这就是为什么只要这张表里第 2 步成功切到 `PENDING`，理论上对应审批人就能看到待办。

## 5. 当前库表与代码的偏差

这部分很重要，因为你要求“所有字段都有用”，实际情况不能这么简单说。

### 5.1 真实情况不是“所有字段都在当前流程中被使用”

当前实际状态是：

- 核心字段在用
- 部分字段是历史兼容字段
- 部分字段在数据库里存在，但实体没映射

## 7. 结合当前线上问题的字段解释

### 7.1 为什么 `zlb_admin` 发起后能推进给 `zlb_audit1`

因为当前修复后，第一步步骤实例会这样变化：

- 第 1 步：
  - `step_name = 填报人提交`
  - `approver_id = requester_id`
  - `status: PENDING -> APPROVED`
  - `comment = 系统自动完成提交流程节点`
- 第 2 步：
  - `status: WAITING -> PENDING`
  - `approver_id = 189`

所以后端数据层已经证明流转成立。

### 7.2 为什么前端仍可能看不到待办

因为前端最终显示待办，不是看 `audit_instance.current_step_index`，而是调用接口查：

- `my-pending?userId=当前用户`

然后后端再按：

- `audit_instance.status = IN_REVIEW`
- `audit_step_instance.status = PENDING`
- `audit_step_instance.approver_id = 当前用户`

返回数据。

如果前端错误地把 `userId` 用成了别的值，比如我们实际抓到的 `userId=1`，那页面就会出现“接口明明有待办、但页面显示 0”的现象。
