# 审批工作流模块化重构设计方案

## 1. 文档目标

本方案用于指导 `sism-backend/sism-workflow` 的模块化重构。

当前已经分别完成以下 4 张核心表的单表重构设计：

- `audit_flow_def`
- `audit_step_def`
- `audit_instance`
- `audit_step_instance`

这 4 份文档已经说明了每张表的目标职责和字段白名单，但还缺少一份面向模块整体的重构书，把以下问题串起来：

- 当前审批主链路到底由哪些表和代码承载
- `sism-workflow` 模块现在为什么难以继续演进
- 未来应该拆成哪些内部子模块
- shared-kernel 与 workflow context 的边界怎么收回
- 重构时先做什么、后做什么，怎样控制风险

本方案只讨论模块设计与代码重构路径：

1. 不直接执行数据库变更
2. 不直接修改生产数据
3. 不一次性推翻现有接口
4. 先收敛领域模型和模块边界，再逐步收敛表字段与查询实现

---

## 2. 重构背景

根据当前数据库与代码排查结果，线上审批主链路真正依赖的核心表只有 4 张：

1. `audit_flow_def`
2. `audit_step_def`
3. `audit_instance`
4. `audit_step_instance`

当前实际流程是：

1. 根据 `flow_code` 或 `entity_type` 选中 `audit_flow_def`
2. 根据 `flow_id` 读取 `audit_step_def`
3. 创建 `audit_instance`
4. 展开 `audit_step_instance`
5. 审批过程中更新实例状态和节点状态
6. 待办查询实际依赖 `audit_step_instance.status = PENDING` 和 `approver_id = 当前用户`

与此同时，模块内部还存在以下结构性问题：

- `sism-workflow` 自身代码很少，但核心审批实体放在 `sism-shared-kernel`
- `WorkflowApplicationService` 同时承担流程解析、步骤展开、审批人解析、状态推进、事件发布
- `WorkflowRepository` 是大而全接口，把模板、实例、任务、分页查询都混在一起
- `workflow_task` / `workflow_task_history` 保留在仓储接口中，但并未进入当前线上审批主链路
- 实例层仍使用 `current_step_index`
- 节点层仍保留 `WAITING` 过渡态
- 步骤定义层的审批规则尚未完全收敛到 `role_id`
- “提交节点自动完成”依赖 `step_name` 是否包含“提交”，属于脆弱规则

这说明现在的主要问题不是“代码不能跑”，而是：

- 模型边界不干净
- 模块职责不清晰
- 代码对历史字段和兼容语义耦合过深
- 后续再加撤回、转办、代理、条件分支、版本化流程时，成本会快速上升

## 3. 当前模块现状判断

## 3.1 现有代码结构

`sism-workflow` 当前大体分为：

- `application`
  - `WorkflowApplicationService`
  - `BusinessWorkflowApplicationService`
  - `ReportWorkflowEventListener`
- `domain`
  - `AuditStatus`
  - `WorkflowRepository`
  - 少量事件定义
- `infrastructure`
  - `WorkflowRepositoryFacade`
  - `JpaWorkflowRepository`
  - `AuditFlowDefJpaRepository`
  - `WorkflowTaskJpaRepository`
- `interfaces`
  - Controller
  - DTO

但真正的核心审批实体并不在这个模块中，而在 `sism-shared-kernel`：

- `AuditFlowDef`
- `AuditStepDef`
- `AuditInstance`
- `AuditStepInstance`
- `WorkflowTask`

这使得 `Workflow Context` 的核心模型没有真正收口到自己的 bounded context 中。

## 3.2 当前实现的主要问题

### 3.2.1 领域对象放错位置

审批流定义、审批实例、节点定义、节点实例都属于工作流上下文核心模型，不应长期放在 shared-kernel。

shared-kernel 应该只保留：

- 聚合根基类
- 领域事件基类
- 通用异常
- 通用工具与跨上下文基础设施

不应承载 workflow context 的业务聚合。

### 3.2.2 应用服务过胖

`WorkflowApplicationService` 当前同时做了：

- 流程模板选择
- 实例启动
- 步骤实例展开
- 审批人解析
- 自动完成提交节点
- 事件发布

这导致：

- 规则无法单独测试
- 以后新增规则只能继续堆在同一个类里
- 应用服务正在承担本应属于领域服务和策略组件的职责

### 3.2.3 仓储接口混合了多个子域

同一个 `WorkflowRepository` 同时负责：

- 流程模板查询
- 实例查询与保存
- 待办查询
- 历史查询
- `workflow_task` 查询

这会带来几个问题：

- 接口含义不稳定
- 基础设施实现过于耦合
- 后续无法按聚合拆分读写模型

### 3.2.4 运行态与定义态边界仍然不干净

当前系统虽然已经以 `audit_flow_def -> audit_step_def -> audit_instance -> audit_step_instance` 为主链路，但代码里仍存在以下问题：

- 实例层保存 `current_step_index`
- 节点实例层存在 `WAITING`
- 模板规则层尚未完全收敛到 `role_id`
- 步骤类型判断仍可能通过 `step_name` 推断

这些都说明“定义态”与“运行态”的职责边界尚未完全完成收敛。

---

## 4. 重构目标

本次模块化重构的目标不是单纯“换个目录”，而是完成以下 6 件事：

1. 把审批核心模型从 `sism-shared-kernel` 收回 `sism-workflow`
2. 以“流程模板定义”和“流程实例运行”拆开模块内部结构
3. 把大应用服务拆成可测试的用例服务 + 领域服务 + 策略组件
4. 把大仓储接口拆成按聚合分组的仓储接口
5. 以当前真实使用的 4 张核心表重建稳定主链路
6. 为后续撤回、转办、代理、版本化、条件扩展留出结构空间

最终目标不是做一个“通用 BPM 引擎”，而是做一个：

- 结构清晰
- 能支撑当前审批业务
- 可以稳定演进
- 能和业务上下文解耦协作

的审批工作流模块。

---

## 5. 重构范围

本次设计范围包括：

- `sism-workflow` 内部分层与职责调整
- 审批核心领域模型归位
- 流程定义、流程运行、业务集成的模块内拆分
- 仓储接口拆分
- 查询接口与待办接口的重组
- 事件发布与业务回调机制重整

本次设计暂不包括：

- 数据库表直接删字段
- 流程分支/并行网关
- 可视化流程编排器
- 完整任务中心引擎化
- 大范围前端改造

---

## 6. 目标模块结构

建议将 `sism-workflow` 内部重构为 3 个核心子模块视角。

## 6.1 Definition 子模块

负责模板定义态。

核心职责：

- 管理流程模板 `AuditFlowDef`
- 管理步骤模板 `AuditStepDef`
- 校验模板合法性
- 提供模板查询、版本选择、启停控制

对应数据：

- `audit_flow_def`
- `audit_step_def`

建议代码结构：

```text
com.sism.workflow.domain.definition
com.sism.workflow.application.definition
com.sism.workflow.infrastructure.persistence.definition
```

核心对象建议：

- `FlowDefinition`
- `StepDefinition`
- `FlowDefinitionRepository`
- `FlowDefinitionQueryService`
- `FlowDefinitionSelectionService`

## 6.2 Runtime 子模块

负责流程运行态。

核心职责：

- 启动审批实例
- 推进审批步骤
- 驳回、撤回、取消
- 管理节点实例
- 提供待办、已办、我发起、历史查询

对应数据：

- `audit_instance`
- `audit_step_instance`

建议代码结构：

```text
com.sism.workflow.domain.runtime
com.sism.workflow.application.runtime
com.sism.workflow.infrastructure.persistence.runtime
```

核心对象建议：

- `AuditInstance`
- `StepInstance`
- `AuditInstanceRepository`
- `StepInstanceRepository`
- `WorkflowRuntimeService`
- `PendingTaskQueryService`

## 6.3 Integration 子模块

负责 workflow 与业务上下文、组织上下文之间的连接。

核心职责：

- 根据组织和角色解析审批人
- 接收业务事件并触发审批
- 审批完成后回调业务模块
- 发布领域事件 / 集成事件

建议代码结构：

```text
com.sism.workflow.domain.integration
com.sism.workflow.application.integration
com.sism.workflow.infrastructure.integration
```

核心对象建议：

- `ApproverResolver`
- `WorkflowStarter`
- `WorkflowResultNotifier`
- `WorkflowBusinessAdapter`

## 7. 目标分层职责

## 7.1 Domain 层

只放审批领域概念和领域规则。

应包含：

- 聚合根
- 实体
- 值对象
- 领域服务接口
- 仓储接口
- 领域事件

不应包含：

- Controller DTO
- JPA 查询拼装
- 用户仓储实现细节
- 业务模块调用编排

## 7.2 Application 层

只负责编排用例，不直接承载复杂审批规则。

建议拆成明确用例：

- `StartWorkflowUseCase`
- `ApproveStepUseCase`
- `RejectStepUseCase`
- `WithdrawWorkflowUseCase`
- `ListMyPendingTasksUseCase`
- `GetWorkflowHistoryUseCase`

应用层负责：

- 事务边界
- 权限上下文透传
- 组装输入输出
- 调用领域服务 / 仓储 / 查询服务

## 7.3 Infrastructure 层

负责：

- JPA Repository
- Query 实现
- 组织与用户数据访问适配
- 事件总线适配
- 外部业务模块适配器

基础设施层不应决定：

- 当前节点如何推进
- 流程模板是否有效
- 驳回后状态如何演化

---

## 8. 目标领域模型

## 8.1 流程定义聚合

聚合根：`FlowDefinition`

职责：

- 维护模板头信息
- 维护步骤定义集合
- 校验步骤顺序、终态、启用状态
- 提供版本与启停规则

内部实体：`StepDefinition`

关键约束：

- `flow_code` 稳定唯一
- `entity_type` 表达模板适用范围
- `step_no` 是正式顺序字段
- `step_type` 决定节点语义
- 模板层审批规则最终收敛到 `role_id`

## 8.2 流程运行聚合

聚合根：`AuditInstance`

职责：

- 表达一次审批流程的整体生命周期
- 表达实例状态
- 驱动节点推进
- 驱动通过、驳回、撤回

内部实体：`StepInstance`

关键约束：

- 实例只保留一个整体状态 `status`
- 当前节点由 `audit_step_instance` 反查，不再长期依赖 `current_step_index`
- 节点状态最终收敛到 `PENDING / APPROVED / REJECTED`
- 每个步骤实例都必须能映射回 `step_def_id`

## 8.3 审批人解析服务

领域服务：`ApproverResolver`

职责：

- 根据 `role_id + requester_org_id + 组织结构上下文`
- 解析当前具体审批人
- 输出运行态需要落库的审批人结果

它不属于模板聚合，也不属于实例聚合，应作为独立领域服务存在。

---

## 9. 目标仓储拆分

当前的 `WorkflowRepository` 需要拆分为按聚合和查询职责划分的接口。

建议拆成：

- `FlowDefinitionRepository`
- `AuditInstanceRepository`
- `StepInstanceQueryRepository`
- `WorkflowTaskReadModelRepository` 或直接删除对应抽象

### 9.1 `FlowDefinitionRepository`

负责：

- 根据 `id` / `flow_code` / `entity_type` 查询模板
- 保存模板定义
- 查询启用版本

### 9.2 `AuditInstanceRepository`

负责：

- 保存实例聚合
- 按实例 ID 查询
- 按业务对象查询活动实例
- 查询我发起的实例

### 9.3 `StepInstanceQueryRepository`

负责：

- 查询我的待办
- 查询我的已办
- 查询实例轨迹
- 查询当前待处理节点

说明：

这些查询本质上更偏读模型，不一定必须挂在聚合仓储上。

### 9.4 关于 `workflow_task`

当前线上主链路没有使用 `workflow_task` / `workflow_task_history`。

本次重构建议：

1. 从主仓储接口中移出 `workflow_task` 相关方法
2. 不再把它作为当前审批主模型的一部分
3. 如果后续要做统一任务中心，再单独设计 read model 或 task 子域

## 10. 目标业务流程

## 10.1 启动审批

目标流程：

1. 根据业务请求识别 `entity_type + entity_id`
2. 通过模板选择器确定可用流程模板
3. 读取模板步骤
4. 创建实例聚合
5. 按模板逐步生成节点实例
6. 提交节点若为 `SUBMIT` 类型，则由规则自动完成
7. 首个真正审批节点置为 `PENDING`
8. 发布 `WorkflowStarted` 事件

与当前实现相比，关键变化是：

- 不再通过 `step_name` 包含“提交”来识别提交节点
- 改为由 `step_type = SUBMIT` 驱动

## 10.2 审批通过

目标流程：

1. 读取实例聚合
2. 锁定当前 `PENDING` 节点
3. 校验当前用户是否有处理权
4. 节点置为 `APPROVED`
5. 若存在下一节点，则激活下一节点为 `PENDING`
6. 若不存在下一节点，则实例置为 `APPROVED`
7. 发布 `WorkflowApproved` / `WorkflowCompleted` 事件

## 10.3 审批驳回

目标流程：

1. 读取实例聚合
2. 锁定当前 `PENDING` 节点
3. 节点置为 `REJECTED`
4. 实例整体置为 `REJECTED`
5. 发布 `WorkflowRejected` 事件

## 10.4 查询待办

目标查询应直接基于节点实例：

- `audit_instance.status = IN_REVIEW`
- `audit_step_instance.status = PENDING`
- `audit_step_instance.approver_id = 当前用户`

实例表只提供整体信息，不负责待办定位。

## 11. 与四张核心表的边界对应

## 11.1 `audit_flow_def`

负责模板头定义：

- `flow_code`
- `flow_name`
- `entity_type`
- `is_enabled`
- `version`

## 11.2 `audit_step_def`

负责步骤模板定义：

- `flow_id`
- `step_no`
- `step_name`
- `step_type`
- `role_id`
- `is_terminal`

## 11.3 `audit_instance`

负责流程运行主事实：

- `flow_def_id`
- `entity_type`
- `entity_id`
- `status`
- `requester_id`
- `requester_org_id`
- `started_at`
- `completed_at`

## 11.4 `audit_step_instance`

负责节点运行事实：

- `instance_id`
- `step_def_id`
- `step_index`
- `step_name`
- `status`
- `approver_id`
- `approver_org_id`
- `comment`
- `approved_at`

## 12. 与其他上下文的协作边界

## 12.1 与 Organization / IAM 的边界

工作流模块不维护用户、角色、组织主数据。

工作流只依赖这些能力：

- 按 `role_id` 查候选用户
- 按组织查当前组织有效审批人
- 获取用户展示名

因此建议通过适配器接口隔离：

- `UserDirectoryGateway`
- `OrganizationGateway`
- `RoleAssignmentGateway`

而不是让领域层直接依赖外部仓储实现。

## 12.2 与 Strategy / Execution 的边界

业务上下文负责：

- 什么时候发起审批
- 审批通过后业务状态如何变化
- 页面上显示哪些业务字段

工作流上下文负责：

- 审批模板选择
- 节点推进
- 审批历史
- 待办与已办

建议采用“业务适配器 + 领域事件”协作：

- 业务模块发起 `StartWorkflowCommand`
- 工作流模块发布 `WorkflowCompletedEvent`
- 业务模块监听后更新业务状态

## 13. 实施阶段建议

本次重构建议按 4 个阶段推进，而不是一次性大改。

> 当前状态说明（2026-03-19）：
>
> - 阶段一：已完成
> - 阶段二：已完成
> - 阶段三：后端读模型收口已完成，数据库未调整
> - 阶段四：后端字段语义收口已完成，数据库字段清理待后续审批通过后再执行

## 阶段一：代码收口，不改行为

目标：

- 新建模块内 definition/runtime/integration 包结构
- 新增分仓储接口
- 将 `workflow_task` 从主链路抽离
- 给当前实现加行为回归测试

这一阶段不追求字段彻底收敛，只追求结构拆分和行为冻结。

当前结果：

- 已完成
- 审批核心模型已迁回 `sism-workflow`
- 旧的大仓储接口与双实现已移除
- 应用层已拆为 query / use case / support 组件

## 阶段二：模型收口，替换脆弱规则

目标：

- 将审批核心实体从 `sism-shared-kernel` 迁入 `sism-workflow`
- 用 `step_type` 替代“名称包含提交”
- 用独立审批人解析服务替代应用服务内嵌逻辑
- 用节点实例反查当前节点，弱化 `current_step_index`

当前结果：

- 已完成
- `step_type` 已取代名称包含“提交”的判断
- `role_id` 已成为审批规则主字段
- 当前节点已优先从节点实例反查

## 阶段三：查询收口，形成稳定读模型

目标：

- 重写待办、已办、历史查询
- 将查询逻辑从聚合仓储中分离
- 明确实例视图和节点任务视图

当前结果：

- 已完成到后端代码层
- 已新增独立读模型服务，实例视图、任务视图、历史视图已从业务门面中拆出
- 待办任务已按节点实例输出，不再使用实例级占位视图
- 数据库结构未变，查询仍基于现有表实现

## 阶段四：字段与表语义收口

目标：

- 代码主模型不再使用冗余字段
- 审批规则统一收敛到 `role_id`
- 为后续数据库清理提供最终证据

当前结果：

- 已完成到后端代码层
- 代码主模型已统一收敛到设计白名单字段
- `AuditStepDef` 主模型已收口到 `step_type + role_id`
- `current_step_index` 已降为兼容镜像字段，主流程不再依赖它定位当前节点
- 运行时主语义已切到 `step_type + role_id`
- 为兼容未完成数据库审批的历史数据，后端保留了旧行归一化逻辑：
  - 缺失 `step_type` 时，临时按步骤名称推断 `SUBMIT/APPROVAL`
  - 缺失 `role_id` 的审批节点，临时回退为发起人审批，并输出告警日志
- 数据库迁移、旧字段下线、历史列清理未执行，待后续数据库审批通过后再推进

## 15. 风险与注意事项

## 15.1 shared-kernel 迁移风险

如果直接移动实体类，容易影响其他模块编译依赖。

建议做法：

1. 先在 `sism-workflow` 建新模型
2. 通过适配层兼容旧引用
3. 最后再删除 shared-kernel 中的 workflow 业务模型

## 15.2 历史字段兼容风险

当前库中仍存在大量历史字段与兼容列。

建议：

- 先停用代码读取
- 再通过数据核验确认是否仍被外部依赖
- 最后再考虑数据库级清理

## 15.3 待办查询风险

待办的真实来源是 `audit_step_instance`，不是 `audit_instance.current_step_index`。

因此任何重构都不能破坏以下事实：

- 待办查询必须以节点实例为准
- 节点状态推进必须严格可追溯

## 15.4 审批人解析风险

审批人解析依赖组织、角色、用户有效性。

这一层必须单独隔离，否则工作流应用服务会持续膨胀。

---

## 16. 最终重构结论

对 `sism-workflow` 的模块化重构，核心结论如下：

1. 当前审批主链路应明确收敛为 `audit_flow_def -> audit_step_def -> audit_instance -> audit_step_instance`。
2. `workflow_task` / `workflow_task_history` 不应继续混入当前审批主仓储与主模型。
3. 审批核心领域模型应从 `sism-shared-kernel` 收回 `sism-workflow`，shared-kernel 只保留真正共享的基础能力。
4. 模块内部应拆成 `definition`、`runtime`、`integration` 三个视角，而不是继续依赖一个胖应用服务加一个大仓储接口。
5. 提交节点识别应由 `step_type` 驱动，不能长期依赖节点名称文本判断。
6. 待办判定与当前节点定位应以 `audit_step_instance` 为准，实例表只保留整体流程事实。
7. 审批人解析应收敛为独立服务，并统一基于模板层 `role_id` 规则。
8. 重构应分阶段推进，先收结构，再收模型，再收查询，最后收字段和数据库语义。

---

## 17. 下一步建议

建议按以下顺序继续推进：

1. 先确认本份模块化重构方案
2. 再基于本方案输出 `sism-workflow` 目录级重构任务拆解
3. 然后落第一阶段代码改造：
   - 包结构调整
   - 仓储接口拆分
   - 查询职责拆分
   - 回归测试补齐

如果这份方案确认通过，下一步就不再停留在文档层，而是可以直接进入 `sism-workflow` 第一阶段代码重构。
