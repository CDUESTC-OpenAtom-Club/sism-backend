# 外置拦截器设计：仅填报人可发起下发审批

## 1. 背景

当前 Plan 下发链路中，前端页面存在“下发 / 提交审批 / 发起审批”入口。  
业务上希望严格限制：

- 只有数据库中具备“填报人”身份的账号，才允许发起下发审批
- 非填报人账号即使能看到页面、点到按钮、或直接调用接口，也不能激活审批流程

本次先输出设计方案，不直接改代码。

## 2. 目标

本方案目标如下：

- 在不改动共享模块的前提下，实现独立、可外挂的权限拦截
- 只限制“发起下发审批”动作，不影响其他查询、审批、撤回等能力
- 权限判断以数据库中的真实角色关系为准，而不是只依赖前端显示或缓存状态
- 对前端和后端都给出可执行的落点建议

## 3. 非目标

本方案暂不处理以下内容：

- 不重构现有角色模型
- 不引入新的“下发人”专用角色
- 不修改 shared 通用鉴权模块
- 不处理历史错误实例数据修复
- 不处理“谁应该显示按钮”的全站权限统一改造

## 4. 现状分析

### 4.1 当前业务入口

Plan 下发审批当前入口为：

- `POST /api/v1/plans/{id}/submit-dispatch`

控制器中该接口最终复用：

- `submitPlanForApproval(...)`

即：

- 页面点击“下发 / 发起审批”
- 进入 `PlanController`
- 调用 `PlanApplicationService.submitPlanForApproval(...)`
- 发布业务事件并启动工作流

这意味着，只要在 `sism-strategy` 模块的控制器层、应用服务层，或该接口链路前面增加一个外置拦截点，就可以锁死“非填报人不能发起”。

### 4.2 当前用户上下文

当前控制器已经可以通过：

- `@AuthenticationPrincipal CurrentUser`

获取当前登录用户信息，至少包括：

- `id`
- `username`
- `orgId`
- `authorities`

因此外挂拦截器时，不需要改 shared 模块来传用户上下文。

### 4.3 当前数据库角色定义

目前 clean seed 中的业务角色为：

- `ROLE_REPORTER`：填报人
- `ROLE_APPROVER`：部门负责人
- `ROLE_STRATEGY_DEPT_HEAD`：战略发展部负责人
- `ROLE_VICE_PRESIDENT`：分管校领导

本次限制的核心判断条件是：

- 当前用户在数据库里是否持有 `ROLE_REPORTER`

## 5. 业务规则

### 5.1 核心规则

发起下发审批时，只有数据库中持有 `ROLE_REPORTER` 的用户可以发起。

### 5.2 拦截结果

- 若当前用户是填报人：允许发起审批
- 若当前用户不是填报人：拒绝发起审批，返回错误

### 5.3 错误文案建议

后端返回：

- HTTP `403 Forbidden`
- message: `当前账号不是填报人，无法发起下发审批`

前端提示：

- `仅填报人可发起审批`

## 6. 设计原则

### 6.1 后端锁死，前端提示

权限控制必须以后端为最终裁决。

原因：

- 前端隐藏按钮不能防止绕过页面直接调接口
- 只有后端才能真正防止审批实例被错误创建

因此建议：

- 后端：强制校验并拒绝非法请求
- 前端：补充按钮禁用或点击提示，提升体验

### 6.2 以数据库为准

本规则应以数据库中的用户-角色关系为准，而不是只信 token 中的 authority。

原因：

- token 可能滞后
- 页面缓存可能滞后
- 业务明确要求“数据库中是谁，谁才有资格发起”

建议校验来源：

- `sys_user_role`
- `sys_role.role_code = 'ROLE_REPORTER'`

## 7. 方案选型

## 7.1 方案 A：前端按钮级限制

做法：

- 前端进入页面后，根据当前用户角色决定是否展示“下发 / 发起审批”按钮
- 非填报人点击时直接 toast 提示

优点：

- 体验好
- 用户一眼能知道自己没有权限

缺点：

- 无法防止直接调用接口
- 不能作为最终权限控制手段

结论：

- 可以做，但只能作为辅助层

## 7.2 方案 B：后端业务接口外置拦截器

做法：

- 不改 shared
- 在 `sism-strategy` 模块单独增加一个拦截器
- 只拦截 `POST /api/v1/plans/{id}/submit-dispatch`
- 进入业务前查询数据库，确认当前用户是否具备 `ROLE_REPORTER`

优点：

- 不影响其他模块
- 与 URL 行为直接对应，语义清晰
- 实现成本低

缺点：

- 只覆盖被配置的接口
- 如果未来新增同类入口，需要同步补充配置

结论：

- 推荐

## 7.3 方案 C：后端 AOP 切面

做法：

- 在 `sism-strategy` 模块对 `PlanController.submitPlanForDispatchApproval(...)` 或 `PlanApplicationService.submitPlanForApproval(...)` 做切面校验

优点：

- 与具体业务方法绑定，不依赖 URL
- 对后续 Controller 路由调整更稳

缺点：

- 可读性不如 URL 拦截器直观
- 若切点选太宽，容易误伤普通提交审批链路

结论：

- 可行，但优先级低于方案 B

## 8. 推荐方案

推荐采用：

- 后端：`sism-strategy` 模块外挂业务拦截器
- 前端：按钮禁用或点击提示

即：

1. 前端尽量不让非填报人误点
2. 后端作为最终闸门锁死

## 9. 推荐落点

### 9.1 后端落点

建议仅在 `sism-strategy` 模块新增如下能力：

- 一个“发起下发审批权限校验器”
- 一个 `HandlerInterceptor` 或等价外置鉴权组件
- 一个本模块自己的 `WebMvcConfigurer` 注册配置

拦截范围建议先只覆盖：

- `POST /api/v1/plans/{id}/submit-dispatch`

如后续确认普通审批入口也需受同样限制，再评估是否扩展到：

- `POST /api/v1/plans/{id}/submit`

### 9.2 前端落点

前端页面若存在“下发 / 发起审批”按钮，建议增加：

- 页面加载时拿当前用户角色
- 若不是填报人，按钮直接禁用或隐藏
- 即使保留可点，也应在点击前给出提示

但前端不承担最终安全责任。

## 10. 权限校验逻辑

建议逻辑如下：

1. 从当前认证上下文获取 `currentUser.id`
2. 查询该用户是否在数据库中绑定 `ROLE_REPORTER`
3. 若否，则立即拒绝请求
4. 若是，则继续进入 `submit-dispatch` 业务流程

建议只判断“是否持有填报人角色”，不额外混入审批人、部门负责人、系统管理员等其他角色语义。

## 11. 为什么不建议改 shared 模块

本需求的边界很明确：

- 只针对 Plan 下发发起动作
- 只影响一个业务模块
- 只是一条细粒度业务规则

如果把它放进 shared，会带来以下风险：

- 共享鉴权语义被业务污染
- 其他模块要被迫理解“填报人才能发起下发审批”这一局部规则
- 后续维护者难以判断这是平台规则还是单业务规则

因此更适合在业务模块本地实现。

## 12. 风险与边界

### 12.1 token 与数据库不一致

如果只靠 token authority，可能出现：

- 数据库已取消填报人角色，但旧 token 仍可用

因此建议后端每次发起审批都查数据库。

### 12.2 一人多角色

若某用户同时具备：

- `ROLE_REPORTER`
- `ROLE_APPROVER`

则本方案仍允许其发起审批。  
因为规则判断的是：

- 是否具备填报人资格

而不是：

- 是否“只有”填报人角色

这点需要业务明确接受。

### 12.3 未来新增“下发人”角色

如果后续角色模型调整为：

- 只有 `ROLE_ISSUER` 才能发起下发

则本方案中的判断条件需从：

- `ROLE_REPORTER`

切换为：

- `ROLE_ISSUER`

当前文档基于现状角色模型，不预埋该变更。

## 13. 接口行为建议

### 13.1 请求成功

- 正常返回 `200`
- 创建或推进审批流程

### 13.2 请求失败

当用户不是填报人时：

- 返回 `403`
- 不创建审批实例
- 不推进任何 workflow
- 不修改 plan 审批状态

## 14. 测试建议

建议至少覆盖以下场景：

### 14.1 正向场景

- `zlb_admin` 发起战略发展部下发审批，允许
- `jiaowu_report` 发起教务处下发审批，允许

### 14.2 反向场景

- `admin` 发起下发审批，拒绝
- `jiaowu_audit1` 发起下发审批，拒绝
- `jiaowu_leader` 发起下发审批，拒绝

### 14.3 绕过验证

- 前端按钮隐藏后，手工调用接口
- 后端仍应拒绝非填报人请求

## 15. 最终结论

在现有架构下，完全可以不改 shared 模块，只在 `sism-strategy` 模块外挂一个拦截层，专门限制：

- 只有数据库中持有 `ROLE_REPORTER` 的用户，才可以调用 Plan 下发审批发起入口

推荐采用：

- 前端做可见性和提示
- 后端做数据库级锁死

这是当前改动面最小、业务边界最清晰、风险最低的方案。
