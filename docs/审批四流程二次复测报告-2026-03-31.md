## 6.2 驳回后重新提交再通过

- 测试 plan：`5044`
- 说明：本节记录的是**重启最新后端前**的旧运行态复测结果，用于解释当时为什么得出“仍有偏差”的结论

6. `zlb_final1` 通过

### 当前真实数据库结果

最终：

- `plan.status = DISTRIBUTED`
- `audit_instance.id = 11`
- `audit_instance.status = APPROVED`

### 驳回后中间节点结果

实例 `11` 的步骤记录如下：

1. `id=36` `step_def_id=7` `填报人提交` -> `APPROVED`
2. `id=37` `step_def_id=8` `职能部门审批人审批` -> `REJECTED`
3. `id=38` `step_def_id=7` `填报人提交` -> `APPROVED`
4. `id=39` `step_def_id=8` `职能部门审批人审批` -> `PENDING/APPROVED`
5. `id=40` `step_def_id=9` `分管校领导审批`
6. `id=41` `step_def_id=10` `战略发展部终审人审批`

### 关键观察

这条链路**没有**表现出“驳回后新增一个 `WITHDRAWN` 的填报人节点，再等待填报人显式重新提交”的效果。

它当前仍然表现为：

1. 原部门审批节点 `REJECTED`
2. 系统自动补出一个新的 `填报人提交` 节点
3. 该新填报人节点直接就是 `APPROVED`
4. 系统立即继续生成新的部门审批节点 `PENDING`

### 结论

此场景最终能够走通并通过，但**行为仍与目标规则不一致**。

该结论仅适用于本次复测早期的旧运行态，不代表重启最新后端后的最终结果；最新结果以 6.3 与第 7 节总结为准。

当前运行中的真实接口行为仍然是：

- 驳回后自动补出新的“已通过填报节点”
- 并直接继续生成下一审批节点

而不是目标规则中的：

- 驳回后先生成一个 `WITHDRAWN` 的填报节点
- 等填报人再次提交后，才生成新的审批节点
