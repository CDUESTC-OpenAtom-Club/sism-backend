# 审批流回退与重提规则说明

## 1. 文档目的

本文档用于统一 SISM 系统中“下发审批”和“上报审批”的节点流转规则，重点说明以下问题：

- 正常通过时，节点如何推进
- 提交后撤回，再次提交时，节点如何恢复
- 审批人驳回后，流程如何回退到上一个业务节点
- 回退后再次提交，节点实例与节点定义应如何对应
- 前端页面应如何展示“当前节点”和“历史节点”

本文档不讨论 UI 样式，只讨论业务规则、数据库节点语义和展示口径。

---

## 2. 适用范围

本文档适用于以下两类流程：

### 2.1 下发审批

- `PLAN_DISPATCH_STRATEGY`
- `PLAN_DISPATCH_FUNCDEPT`

### 2.2 上报审批

- `PLAN_APPROVAL_FUNCDEPT`
- `PLAN_APPROVAL_COLLEGE`

这四类流程虽然审批层级不同，但都应遵循同一套“提交 / 撤回 / 驳回 / 重提”的回退原则。

---

## 3. 基本概念

## 3.1 流程定义节点

流程定义节点来自 `audit_step_def`。

例如 `PLAN_DISPATCH_FUNCDEPT` 当前 canonical 定义为：

1. `step_def_id = 4` `填报人提交`
2. `step_def_id = 5` `职能部门审批人审批`
3. `step_def_id = 6` `分管校领导审批`

无论实例如何回退、撤回、重提，实例步骤都必须仍然映射到这些既有定义节点，而不是跳到新的业务含义节点。

## 3.2 流程实例节点

流程实例节点来自 `audit_step_instance`。

说明：

- `step_def_id` 表示该实例步骤对应哪个流程定义节点
- `step_no` 表示该条实例记录的顺序号
- `status` 表示该实例步骤当前状态

当前系统允许一个 `audit_instance` 下存在多条映射到同一个 `step_def_id` 的实例步骤，用于表达“重走一轮”。

## 3.3 本文中的核心约束

本次规则统一后的核心要求如下：

1. 回退后新增的实例节点，必须仍然对应原流程定义节点。
2. 驳回后不能直接自动跳过填报人重新提交动作。
3. 撤回后的恢复，应恢复到原业务节点，而不是重新造一套新业务节点语义。
4. 前端展示时，应按“流程定义节点”理解，不应把每次重走都渲染成全新业务阶段。

---

## 4. 节点状态语义

建议统一使用以下语义理解：

- `PENDING`
  当前节点待当前角色处理。

- `WAITING`
  当前节点不是活动节点，只是等待前置节点重新完成后才能恢复。

- `APPROVED`
  该节点本轮已通过。

- `REJECTED`
  该节点本轮被驳回。

- `WITHDRAWN`
  该节点所在的本轮提交被撤回，或者该节点被回退为“等待重新提交”的占位节点。

说明：

- 在“发起人主动撤回”场景中，`WITHDRAWN` 是明确成立的。
- 在“审批人驳回并退回填报人”场景中，是否继续复用 `WITHDRAWN` 作为“待重新提交”状态，需要前后端统一；本文建议允许复用，但必须保证前端文案明确区分“撤回”与“退回待重提”。

---

## 5. 三类基础场景

下面以 `PLAN_DISPATCH_FUNCDEPT` 为例说明，其他流程同理，只是审批链长度不同。

---

## 5.1 场景 A：正常通过

### 业务过程

1. 填报人提交
2. 部门审批人通过
3. 分管校领导通过
4. 流程结束

### 节点变化

1. `填报人提交` -> `APPROVED`
2. `职能部门审批人审批` -> `PENDING`
2. `职能部门审批人审批` -> `APPROVED`。
3. `分管校领导审批` -> `PENDING`
3. `分管校领导审批` -> `APPROVED`
结束

### 业务状态变化

- `audit_instance.status` 最终变为 `APPROVED`
- `plan.status` 或对应业务状态最终变为“已下发 / 已通过”

---

## 5.2 场景 B：提交后撤回，再次提交，再通过

### 业务过程

1. 填报人提交
2. 系统进入部门审批节点
3. 发起人撤回
4. 发起人再次提交
5. 后续正常审批通过

### 目标节点变化

1. `填报人提交` -> `APPROVED`
2. `职能部门审批人审批` -> `PENDING`
3. 发起人撤回后：
   `填报人提交` -> `WITHDRAWN`
   `职能部门审批人审批` -> `WAITING`
4. 再次提交后：
   `填报人提交` -> `APPROVED`
   `职能部门审批人审批` 恢复为 `PENDING`
5. 后续按正常链路走完

### 规则说明

- 撤回场景优先理解为“恢复原链路”
- 不要求新增新的业务定义节点
- 可以保留同一实例中的历史记录，但恢复后的当前节点必须仍是原定义节点

---

## 5.3 场景 C：部门审批人驳回，然后填报人再提交，再通过

这是当前最容易引发歧义的场景，也是本文重点。

### 业务过程

1. 填报人提交
2. 部门审批人驳回
3. 流程回退到填报人
4. 填报人修改后重新提交
5. 部门审批人再次审批
6. 后续继续通过

### 目标节点变化

第一轮：

1. `节点 1 (填报人提交)` -> `APPROVED`
2. `节点 2 (职能部门审批)` -> `PENDING`
3. `节点 2 (职能部门审批)` -> `REJECTED`

回退后（第二轮重提过程）：

4. **产生新的节点 1 记录**：`step_def_id = 1 (填报人提交)` -> **状态置为 `WITHDRAWN`**。
   * **业务关联**：此时 `plan` 业务表状态回退至 `DRAFT`（草稿），系统应允许填报人进行编辑修改。
5. **用户再次操作**：填报人编辑完成后，再次点击“**下发**”按钮。
6. **节点 1 完成转义**：该节点 1 的实例状态由 `WITHDRAWN` 变为 **`APPROVED`**。
7. **产生新的节点 2 记录**：系统紧接着生成新一轮的 `step_def_id = 2 (职能部门审批)`，初始状态为 **`PENDING`（待审批）**。

### 本场景的核心规则

1. 驳回后回到的必须是“填报人提交”这个定义节点，且状态标记为 `WITHDRAWN`（填报人专属状态）。
2. 在填报人未点击“下发”前，系统禁止自动生成后续的节点 2 审批记录。
3. 只有填报人确认重提（Node 1 转为 `APPROVED`），下一步的 `PENDING` 才会出现。

### 不应出现的行为

以下行为不符合本文规则：

1. **[逻辑 2 驳回 1] 后自动跳步**：职能部门（2）驳回后，系统直接将新的节点 1 设为 `APPROVED`。**正确逻辑**：节点 1 必须停留在 `WITHDRAWN`。
2. **[逻辑 3 驳回 2] 后未等待重提**：校领导（3）驳回后回到职能部门（2），系统自动通过了节点 2 的审批。**正确逻辑**：节点 2 必须由审批人重新手动操作。
3. **跨逻辑跳步**：在节点 1 尚未完成（未 `APPROVED`）时，系统跳过顺序直接生成节点 3 的记录。

---

## 5.4 典型“错误说法”与正确逻辑对比

基于 1（填报）、2（部门）、3（校办）的固定序列：

| 节点位置 | 正确逻辑 (1-2-3 推进) | 常见的错误说法/实现 |
| :--- | :--- | :--- |
| **节点 1** | 驳回后产生 `WITHDRAWN` 记录，对应 `DRAFT` 状态。点击下发后才变 `APPROVED`。 | “驳回系统自动提交”，系统自动帮用户完成节点 1 的新记录生成并 `APPROVED`。 |
| **节点 2** | 仅在收到节点 1 的 `APPROVED` 信号后，才创建新的 `PENDING` 记录。 | “驳回后直接待审”，节点 1 还没提交，节点 2 就已经出现了新的 `PENDING`。 |
| **节点 3** | 只有节点 2 完成新的 `APPROVED`，节点 3 才会收到推送。 | “跳过环节”，驳回重提后直接绕过 2 的再次审批到达 3。 |

## 6. 对“新增实例节点”的统一理解

为了兼容历史审计，系统可以允许同一个 `audit_instance` 中出现多条映射到同一 `step_def_id` 的步骤实例。

例如：

- 第一次的 `填报人提交`
- 驳回后的第二次 `填报人提交`

这两条都可以存在，但必须满足：

1. 都映射到同一个定义节点
2. 前端主流程展示按定义节点理解，而不是按纯实例行数理解
3. 历史中应能看出第几轮被驳回、第几轮重新提交

也就是说：

- “允许新增实例记录”是可以的
- “不允许新增新的业务节点语义”是必须的

---

## 7. 前端展示规则

## 7.1 主流程展示

前端审批中心主视图应优先按“流程定义节点”展示，例如：

1. 填报人提交
2. 职能部门审批人审批
3. 分管校领导审批

而不是直接按数据库实例的 `step_no = 1,2,3,4,5...` 全量平铺。

## 7.2 历史展示

历史区域可以展示“第几轮”：

- 第一轮：部门审批驳回
- 第二轮：填报人重新提交
- 第二轮：部门审批通过

这样既不丢历史，又不会把主流程展示成越来越长。

## 7.3 状态文案

建议展示文案：

- `PENDING` -> `待审批`
- `WAITING` -> `等待前置节点`
- `APPROVED` -> `已通过`
- `REJECTED` -> `已驳回`
- `WITHDRAWN` -> `已撤回` 或 `待重新提交`

说明：

如果后端决定在“驳回回退到填报人”场景继续复用 `WITHDRAWN`，前端应结合上下文将其展示为“待重新提交”，避免误解为“主动撤回”。

---

## 8. 后端实现建议

## 8.1 撤回场景

撤回可以继续采用“恢复原链路”的模式：

- 当前提交节点记为 `WITHDRAWN`
- 后续待审批节点记为 `WAITING`
- 重新提交时恢复当前链路

## 8.2 驳回场景

驳回场景建议采用以下规则：

1. 当前审批节点变为 `REJECTED`
2. 生成一个新的“上一个定义节点”的实例记录
3. 若该上一个节点是填报人提交，则该节点应停留在“待重新提交”状态
4. 只有用户再次提交后，该节点才变为 `APPROVED`
5. 然后再生成新的下一审批节点

## 8.3 应避免的实现

当前代码若存在以下行为，应视为待修复：

1. 驳回后立即把新的提交节点自动 `APPROVED`
2. 不等待用户重新提交，就自动生成新的审批节点
3. 使用 `step_no` 直接代表业务阶段，导致一轮轮重走后主流程越来越长

---

## 9. 建议补充的测试场景

除了当前已有的正常通过测试，还应补以下场景：

### 9.1 提交后撤回，再提交

验证：

- 提交节点由 `APPROVED -> WITHDRAWN -> APPROVED`
- 下一审批节点由 `PENDING -> WAITING -> PENDING`

### 9.2 部门审批驳回后，填报人未重新提交

验证：

- 驳回节点应为 `REJECTED`
- 新的填报节点应处于“待重新提交”状态
- 不应自动生成新的部门审批 `PENDING`

### 9.3 部门审批驳回后，填报人重新提交

验证：

- 新填报节点提交后变为 `APPROVED`
- 之后才生成新的部门审批节点 `PENDING`

### 9.4 校领导驳回

验证：

- 校领导节点变为 `REJECTED`
- 回退到上一个定义节点
- 上一个定义节点重新进入可处理状态

### 9.5 多轮驳回 / 多轮重提

验证：

- 每一轮新增的实例节点都必须仍然映射到原 `step_def_id`
- 前端主流程仍然只显示定义节点，不显示无限增长的业务阶段

---

## 10. 结论

本文档最终确认以下原则：

1. 系统允许保留历史实例步骤。
2. 回退后可以新增新的实例步骤记录。
3. 但新增记录必须仍然对应原有流程定义节点。
4. 驳回回退到填报人后，不应自动替用户完成重新提交。
5. 前端主视图应按“定义节点”展示，历史区域再展示多轮执行记录。

这套规则同时适用于下发审批与上报审批。

后续无论修数据库逻辑、服务端同步逻辑，还是审批中心展示逻辑，都应以本文档为准。
