# 认证授权模块迁移文档> 迁移日期：2026-03-12> 任务：1.2.2 迁移认证授权模块## 迁移概述将认证授权模块从传统的 `com.sism` 包结构迁移到新的 DDD 架构的 `com.sism.shared` 包结构。## 迁移内容### 1. AuthController 迁移**原位置**: `com.sism.controller.AuthController`**新位置**: `com.sism.shared.interfaces.rest.AuthController`**功能**:- POST `/api/auth/login` - 用户登录- POST `/api/auth/logout` - 用户登出- POST `/api/auth/refresh` - 刷新令牌- GET `/api/auth/userinfo` - 获取用户信息**特性**:- JWT 访问令牌（15分钟有效期）- HttpOnly Cookie 存储刷新令牌（7天有效期）- 令牌轮换机制- CSRF 保护（SameSite=Strict）### 2. JWT 服务实现**新文件**: `com.sism.shared.application.auth.JwtServiceImpl`**功能**:- 生成访问令牌和刷新令牌- 解析和验证 JWT 令牌- 提取用户信息（userId, username, orgId）**配置**:- `jwt.secret`: JWT 签名密钥- `jwt.expiration`: 访问令牌有效期（默认 15 分钟）- `jwt.refresh-expiration`: 刷新令牌有效期（默认 7 天）### 3. RefreshToken 服务**接口**: `com.sism.shared.application.auth.RefreshTokenService`**实现**: `com.sism.shared.application.auth.RefreshTokenServiceImpl`**功能**:- 生成和验证刷新令牌- 令牌轮换（Token Rotation）- 撤销令牌- 会话管理（最多 5 个并发会话）- 定期清理过期令牌**安全特性**:- 使用 SecureRandom 生成随机令牌- 存储 SHA-256 哈希值- 优雅降级（数据库表不存在时自动禁用）### 4. AuthService 实现**接口**: `com.sism.shared.application.auth.AuthService`**实现**: `com.sism.shared.application.auth.AuthServiceImpl`**功能**:- 用户登录验证- 令牌生成和刷新- 用户信息获取- 令牌黑名单管理### 5. Spring Security 配置**位置**: `com.sism.shared.infrastructure.security.SecurityConfig`**配置内容**:- JWT 认证过滤器链- CORS 配置（支持本地开发）- 公开端点配置- 无状态会话管理**公开端点**:- `/api/auth/**` - 认证接口- `/api/health/**` - 健康检查- `/swagger-ui/**` - API 文档- 其他业务端点（临时开放，后续需要权限控制）### 6. JWT 认证过滤器**位置**: `com.sism.shared.infrastructure.security.JwtAuthenticationFilter`**功能**:- 从 Authorization 头提取 JWT 令牌- 验证令牌有效性- 检查令牌黑名单- 设置 Spring Security 上下文## 向后兼容性### 保留的旧代码以下旧代码暂时保留，以确保向后兼容：1. `com.sism.controller.AuthController` - 旧的认证控制器2. `com.sism.service.AuthService` - 旧的认证服务3. `com.sism.service.RefreshTokenService` - 旧的刷新令牌服务4. `com.sism.config.SecurityConfig` - 旧的安全配置5. `com.sism.util.JwtUtil` - 旧的 JWT 工具类### 迁移策略- **阶段 1**: 新代码与旧代码并存（当前阶段）- **阶段 2**: 逐步将依赖切换到新的 shared 包- **阶段 3**: 删除旧代码## API 端点变更### 新端点路径所有认证端点统一使用 `/api/auth` 前缀：- `/api/auth/login` - 登录- `/api/auth/logout` - 登出- `/api/auth/refresh` - 刷新令牌- `/api/auth/userinfo` - 获取用户信息### 旧端点路径旧的 `/auth/*` 路径仍然可用，但建议使用新的 `/api/auth/*` 路径。## 数据库依赖### refresh_tokens 表刷新令牌功能依赖 `refresh_tokens` 表：```sqlCREATE TABLE refresh_tokens (    id BIGSERIAL PRIMARY KEY,    user_id BIGINT NOT NULL,    token_hash VARCHAR(64) NOT NULL,    expires_at TIMESTAMP NOT NULL,    revoked_at TIMESTAMP,    device_info VARCHAR(500),    ip_address VARCHAR(45),    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,    CONSTRAINT fk_refresh_token_user FOREIGN KEY (user_id) REFERENCES sys_users(id));```**优雅降级**: 如果表不存在，刷新令牌功能会自动禁用，但登录功能仍然可用。## 配置参数### application.yml```yamljwt:  secret: sism-strategic-indicator-management-system-secret-key-2024  expiration: 900000  # 15 minutes in milliseconds  refresh-expiration: 604800000  # 7 days in milliseconds  max-sessions-per-user: 5  # Maximum concurrent sessions per user```## 测试### 现有测试以下测试仍然针对旧的实现：1. `AuthServiceTest` - 认证服务单元测试2. `AuthControllerIntegrationTest` - 认证控制器集成测试3. `SecureFunctionalityPreservationTest` - 安全功能保留测试### 测试策略- 旧测试继续运行，确保旧代码功能正常- 后续需要为新的 shared 包实现添加测试## 编译验证```bashcd sism-backendmvn clean compile -DskipTests```**结果**: ✅ 编译成功，无错误## 下一步1. 更新前端代码，使用新的 `/api/auth/*` 端点2. 为新的 shared 包实现编写单元测试和集成测试3. 逐步将其他模块的依赖切换到新的 shared 包4. 验证所有功能正常后，删除旧代码## 注意事项1. **Cookie 安全**: 刷新令牌使用 HttpOnly、Secure、SameSite=Strict 属性2. **令牌轮换**: 每次刷新令牌时，旧令牌会被撤销，生成新令牌3. **会话限制**: 每个用户最多 5 个并发会话，超过时自动撤销最旧的会话4. **定时清理**: 每天凌晨 3 点自动清理过期和已撤销的令牌5. **IP 追踪**: 记录每个令牌的 IP 地址和设备信息，用于安全审计## 相关文档- [架构重构需求文档](../../.kiro/specs/architecture-refactoring/requirements.md)- [架构重构设计文档](../../.kiro/specs/architecture-refactoring/design.md)- [架构重构任务清单](../../.kiro/specs/architecture-refactoring/tasks.md)
