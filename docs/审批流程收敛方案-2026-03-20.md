# 审批流程收敛方案

## 1. 背景与目标

当前审批流程已经具备模板定义、实例启动、节点审批、节点驳回等基础能力，但在产品理解、代码实现和客户预期之间还存在差异，主要体现在：

- 对“是否允许运行时临时插节点”存在误解
- 审批实例状态与业务状态表达不完全统一
- 模板规则尚未被严格约束
- 前端、后端、客户对“退回到最初节点是否结束流程”表述不一致

本方案的目标是把审批流程能力明确收敛为一套固定规则：

- 模板先定义，实例后运行
- 运行中不允许改模板结构
- 流程只允许前进和后退
- 最后一个节点通过后整体结束
- 退回到最初提交节点时流程不结束，而是等待重新提交

- 发起时为审批节点选择具体审批人

- 节点驳回后退回到上一个已通过节点
- 最后一个节点通过后，审批实例整体完成

### 2.2 现状问题

当前实现存在以下待收口问题：

- `WorkflowStatus` 仍保留旧的指标生命周期语义，容易和审批实例状态混淆
- 模板首节点是否为提交节点，目前部分依赖兼容推断，而不是硬性校验
- 运行时“转办/加审批人”接口还是占位实现，不适合被解释为支持动态改流程
- 是否允许动态插节点没有产品与技术共识，但当前系统并未真正支持

### 2.3 当前真实结论

基于现有代码，当前系统更适合被正式定义为：

- 固定模板驱动的线性审批流
- 不支持运行时动态插节点
- 不支持分支跳转
- 不支持并行会签
- 不支持运行中重排节点

## 3. 业务规则收敛

### 3.1 模板规则

模板采用数据库静态定义，运行中的流程实例严格按照模板流转。

规则如下：

1. 每个审批流程必须对应一个唯一模板。
2. 每个模板必须有明确的 `flow_code`、`entity_type`、启用状态和版本号。
3. 每个模板必须按 `step_no` 顺序定义节点。
4. 每个模板必须以一个 `SUBMIT` 类型节点作为首节点。
5. 除首节点外，其余节点必须全部是 `APPROVAL` 类型节点。
6. 所有 `APPROVAL` 节点必须配置角色约束 `role_id`。
7. 模板一旦被实例使用，运行中的实例不允许因“领导临时要求”而动态插入新节点。

### 3.2 运行规则

流程运行规则收敛如下：

1. 发起时选择模板。
2. 前端仅允许为模板中的审批节点选择具体审批人。
3. 启动实例后，首个 `SUBMIT` 节点自动完成。
4. 当前节点审批通过时，流程推进到下一个节点。
5. 如果当前节点是最后一个审批节点，审批通过后实例整体完成。
6. 当前节点审批驳回时，流程退回到上一个已通过节点。
7. 如果退回到首个 `SUBMIT` 节点，流程不结束，而是等待发起人重新提交。
8. 流程不支持跨级跳转，不支持绕过中间节点。
9. 流程不支持“突然增加一级校领导审批”这种运行时结构变更。

### 3.3 明确不支持的能力

为避免范围蔓延，本轮明确不支持：

- 运行中动态插节点
- 条件分支
- 并行会签
- 运行中删除某个审批节点
- 运行中修改模板顺序
- 运行中把整条流程切换成另一模板

如果未来真的出现“需要额外加审”的场景，建议采用：

- 新建模板版本
- 或为该业务单独设计另一条模板

不建议在现有线性流程实例中做运行时结构变更。

## 4. 状态模型建议

### 4.1 审批实例状态

审批实例对外状态建议统一为：

- `IN_REVIEW`
- `APPROVED`
- `REJECTED`

说明：

- `IN_REVIEW` 表示流程正在审批中，包含“待当前节点处理”的含义
- `APPROVED` 表示所有审批节点已通过，实例结束
- `REJECTED` 表示当前轮流程被打回到不可继续状态

发起人撤回不作为独立对外审批状态展示。若首个审批节点尚未开始处理，发起人可撤回并结束当前流程实例，同时业务回到“未下发/可修改”状态。

### 4.2 节点状态

节点状态建议统一为：

- `WAITING`
- `PENDING`
- `APPROVED`
- `REJECTED`

说明：

- `WAITING` 表示尚未轮到该节点
- `PENDING` 表示当前节点待处理
- `APPROVED` 表示该节点已通过
- `REJECTED` 表示该节点在本轮被驳回

### 4.3 业务状态与审批状态关系

对于 Plan 下发流程，建议统一为：

1. 提交审批后，业务状态进入“审批中”语义。
2. 审批实例完成前，不应提前变成“已下发”。
3. 最后一个节点通过后，业务状态再更新为“已下发”。
4. 驳回后业务状态回到可编辑/待重新提交语义。

## 5. 接口规划

## 5.1 设计原则

客户希望操作简单，建议把业务动作收敛为两类：

- 提交流程
- 审核流程

底层仍保留通用工作流接口，但对业务层的公开语义要更清晰。

### 5.2 推荐接口划分

#### A. 模板预览

用途：前端选择模板后，展示节点列表和每个节点可选审批人。

建议接口：

- `GET /api/v1/workflows/definitions/code/{flowCode}/preview`

返回内容：

- 模板编码
- 模板名称
- 节点顺序
- 节点类型
- 节点角色
- 候选审批人列表

#### B. 提交流程

用途：业务发起审批实例。

建议接口：

- `POST /api/v1/workflows/start`

建议请求体包含：

- `workflowCode`
- `businessEntityType`
- `businessEntityId`
- `selectedApprovers`

其中 `selectedApprovers` 表示每个审批节点选定的用户。

#### C. 审核通过

建议接口：

- `POST /api/v1/workflows/tasks/{taskId}/approve`

请求体：

- `comment`

#### D. 审核驳回

建议接口：

- `POST /api/v1/workflows/tasks/{taskId}/reject`

请求体：

- `reason`

#### E. 撤回流程

建议接口：

- `POST /api/v1/workflows/{instanceId}/cancel`

### 5.3 对业务接口的包装建议

对客户和前端可以进一步包装成业务语义：

- `提交审批`
- `审批通过`
- `审批驳回`
- `撤回审批`

避免直接暴露过多工作流引擎概念。

### 5.4 最小可跑通链路

如果当前目标是先快速联调成功，建议只走下面 3 个接口：

1. `GET /api/v1/workflows/definitions/code/{flowCode}/preview`
   - 目的：拿到模板步骤和每个审批节点的候选审批人
2. `POST /api/v1/workflows/start`
   - 目的：按模板编码创建 `audit_instance` 和 `audit_step_instance`
   - 关键字段：`workflowCode`、`businessEntityType`、`businessEntityId`、`selectedApprovers`
3. `POST /api/v1/workflows/tasks/{taskId}/approve`
   - 目的：审批当前 `PENDING` 节点并推进到下一个节点

如果前端更习惯先拿模板 ID 再启动，也可以用：

- `POST /api/v1/workflows/{definitionId}/instances`

这条接口现在也应与 `/start` 保持同样能力，支持把 `selectedApprovers` 一并传入，避免“按模板编码可选人、按模板 ID 启动却丢失选人结果”的割裂。

## 6. 后续改造清单

### 6.1 必须优先完成

1. 收口状态定义，避免 `WorkflowStatus` 和实例状态语义冲突。
2. 对模板增加硬校验：
   - 首节点必须为 `SUBMIT`
   - 后续节点必须全为 `APPROVAL`
   - 审批节点必须配置 `role_id`
   - `step_no` 必须连续且唯一
3. 明确文档中“不支持动态插节点”。
4. 确保 Plan 在最后一个节点通过前不会提前进入“已下发”。

### 6.2 建议随后完成

1. 区分“业务状态”和“审批状态”的展示字段。
2. 在前端模板预览页面明确显示：
   - 节点顺序
   - 节点角色
   - 当前选定审批人
3. 为“退回到提交节点后重新提交”补全端到端回归测试。

### 6.3 暂不纳入本轮

1. 删除整个流程实例
2. 动态插入额外审批节点
3. 条件路由
4. 并行节点
5. 多分支汇合

## 7. 推荐的最终对外表述

建议统一对内对外口径如下：

> 审批流程采用数据库中预先定义好的固定模板执行。前端发起时只负责选择模板，并为模板中的审批节点选择对应审批人。流程启动后，严格按照模板顺序流转，只允许前进和退回，不支持运行中动态增删节点，也不支持分支跳转。最后一个节点审批通过后，流程整体结束，业务状态更新为已下发。若流程被退回到最初提交节点，流程不结束，而是等待重新提交。

## 8. 复现与验收方案

以下方案用于后续联调和回归验证。

### 8.1 前置条件

1. 后端服务启动在 `http://localhost:8080`
2. 前端服务启动在 `http://localhost:3500`
3. 数据库中已存在目标模板，例如 `PLAN_DISPATCH_STRATEGY`
4. 模板满足如下结构：
   - 第 1 节点：`SUBMIT`
   - 第 2 节点：一级审批
   - 第 3 节点：二级审批
5. 模板中每个审批节点角色均能查到候选用户

### 8.2 正向通过场景

1. 在前端选择某个 Plan 下发审批模板。
2. 为模板中的一级、二级审批节点分别选择审批人。
3. 提交审批，系统创建审批实例。
4. 发起节点自动完成，流程进入一级审批节点。
5. 一级审批人登录后点击“审批通过”。
6. 流程推进到二级审批节点。
7. 二级审批人登录后点击“审批通过”。
8. 流程整体完成，Plan 业务状态更新为“已下发”。

预期结果：

- `audit_instance.status = APPROVED`
- 最后一个 `audit_step_instance.status = APPROVED`
- Plan 业务状态为 `DISTRIBUTED`

### 8.3 驳回回退场景

1. 重复 8.2 的前 1 到 5 步。
2. 二级审批人点击“审批驳回”。
3. 流程退回到上一级已通过节点。
4. 上一级重新处理后，可再次进入二级审批。

预期结果：

- 实例不应越级跳转
- 被驳回节点后续节点应恢复为 `WAITING`
- 上一级节点恢复为 `PENDING`

### 8.4 退回到最初提交节点场景

1. 发起审批后，在第一个审批节点执行驳回。
2. 系统应退回到 `SUBMIT` 节点语义。
3. 发起人重新提交后，再次进入第一审批节点。

预期结果：

- 流程不应被删除
- 流程不应进入最终完成态
- 发起人可以重新提交

### 8.5 不允许动态插节点验证

1. 启动一个已在审批中的实例。
2. 尝试通过前端或后端接口新增一个中间审批节点。

预期结果：

- 系统无此能力
- 实例节点顺序保持不变
- 只能按模板原顺序流转

## 9. 结论

本轮建议不再继续讨论“运行时临时插节点”，而是正式把审批流程定义为“固定模板驱动的线性审批流”。这既符合客户要求，也更符合当前系统实际实现，能够以最小风险把审批流先稳定下来。
