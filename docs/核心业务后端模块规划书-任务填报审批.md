# 核心业务后端模块规划书（任务、填报、审批）

> 版本：v1.0
> 日期：2026-03-20
> 依据：`database/seeds` 当前干净种子数据
> 适用范围：`sism-strategy`、`sism-task`、`sism-execution`、`sism-workflow`、`sism-alert`、`sism-iam`、`sism-organization`

## 1. 规划目标

本规划书不是重新发明一套新业务，而是基于你们当前已经落到数据库里的核心表和字段，把后端逻辑重新收敛成一套更清晰、更容易落代码的模块边界。

本次规划重点解决 5 个问题：

1. 任务、填报、审批三条主线目前交叉较多，需要重新明确谁是主模块
2. `plan`、`sys_task`、`indicator`、`plan_report` 的职责已经基本成型，但语义还没有完全统一
3. 当前审批主链路完整覆盖了 `PLAN`，但还没有完整覆盖 `plan_report`
4. 审批节点已经采用合并角色模型，后端必须用“节点上下文 + 组织上下文”路由审批人
5. `workflow_task` 适合做待办投影，不适合继续作为审批事实主表

本规划书的目标结论是：

- 以“计划包下发”与“月度填报执行”两条主链路重构后端
- 让 `task` 成为任务中心，让 `execution` 成为填报中心，让 `workflow` 成为审批中心
- 明确哪些表是主事实表，哪些表只是投影、附件、预警、历史
- 为后续代码重构、接口重整、权限收敛提供统一基线

## 2. 基于当前种子数据的业务事实

根据 `database/seeds` 当前数据与本轮业务修正，核心业务链路应按下面口径理解：

```text
Cycle -> FunctionalPlan -> FunctionalTask -> Indicator -> IndicatorMilestone
                                                |
                                                -> CollegeUndertakeIndicator
                                                -> PlanReport -> PlanReportIndicator -> Attachment
Plan / PlanReport -> AuditInstance -> AuditStepInstance
AuditInstance -> WorkflowTask(待办投影)
PlanReportIndicator -> AlertEvent
```

这里要特别说明一条已经确认的业务修正：

- 任务只存在于“战略发展部 -> 职能部门”这一层
- 二级学院没有独立 `sys_task`
- 二级学院承接的是“职能部门任务下拆分后的指标”，不是新的任务
- 因此当前种子里学院任务样例只应视为临时错误样例，不应作为目标模型继续沿用

结合现有表，建议将业务概念统一解释如下：

### 2.1 周期层

- `cycle`
- 表示年度考核周期
- 是所有计划、任务、填报、预警的时间边界

### 2.2 计划包层

- `plan`
- 不是“单个任务”，而是“面向某个目标组织的一整个年度业务包容器”
- 当前种子已经体现：
  - 战略发展部向职能部门下发 `STRAT_TO_FUNC`
  - 职能部门向学院下发 `FUNC_TO_COLLEGE`
- 但这里的 `FUNC_TO_COLLEGE` 不应再理解为“给学院生成新的任务”
- 它更适合被理解为“职能部门把本部门任务下的指标拆分并下达到学院的执行包/承接包”
- `plan.status` 是当前最权威的包级业务状态

### 2.3 任务层

- `sys_task`
- 是“战略发展部下发给职能部门”的业务任务容器
- 不应在二级学院层重复创建一套学院任务
- 当前一条职能部门 `plan` 下面通常有一条代表性任务，但从模型上应该允许一对多
- `task_type` 已经表明任务中心未来可以扩展，不应把它继续做成只服务战略模块的一张从表

### 2.4 指标层

- `indicator`
- 是真正执行和考核的颗粒
- `owner_org_id` 表示下发/拥有方
- `target_org_id` 表示承接/执行方
- `responsible_user_id` 表示当前责任人
- 对学院场景，学院承接的是职能部门任务下拆分出来的指标
- 因此学院执行链路应以 `indicator.target_org_id = 学院` 为核心，而不是以学院 `task_id` 为核心
- `indicator.status` 目前是指标自身的生命周期投影字段，不应替代 `plan.status`

### 2.5 里程碑层

- `indicator_milestone`
- 是指标的阶段目标，不归属于 `plan` 或 `task`
- 应继续保持挂靠 `indicator_id` 的单一归属，不要额外补 `plan_id`

### 2.6 填报层

- `plan_report`
- 是月度执行填报主表，不是分析报表
- `plan_report_indicator`
- 是对每个指标在该月份的填报事实
- `attachment` + `plan_report_indicator_attachment`
- 是填报证明材料

### 2.7 审批层

- `audit_flow_def`
- `audit_step_def`
- `audit_instance`
- `audit_step_instance`

这 4 张表已经构成当前真实审批主链路。

当前种子明确说明：

- 审批模板主要针对 `PLAN`
- `plan_report` 还没有独立审批实例链
- `workflow_task` 只是轻量待办投影

### 2.8 预警层

- `alert_rule`
- `alert_event`
- 预警来源已经与 `plan_report_indicator` 形成闭环
- 说明预警应该被视为“填报后的衍生能力”，而不是独立主业务域

## 3. 当前系统的核心问题

基于这些表结构和样例数据，当前后端最大的结构问题不是表设计错误，而是模块语义还没有彻底对齐。

### 3.1 `plan` 与 `plan_report` 分属两个阶段，但代码上容易混成一个概念

应严格区分：

- `plan`
  - 下发包
  - 年度目标容器
  - 关注“谁下发给谁、是否审批通过、是否正式生效”
- `plan_report`
  - 月度执行记录
  - 关注“本月填了什么、提交了没有、审批通过没有”

如果这两个概念继续混在一起，后面接口、审批、状态流都会持续打架。

### 3.2 任务模块与规划模块边界仍然模糊

当前数据库上看：

- `plan` 负责包
- `sys_task` 负责任务
- `indicator` 负责执行指标

结合本轮确认的真实业务口径，还需要再补一条：

- `sys_task` 只属于职能部门层
- 二级学院只承接指标，不拥有独立任务

这说明 `Task` 不应该承接规划审批主导权，也不应该拥有填报事实；它应回归任务中心，并且任务中心的边界应明确停在职能部门层。

### 3.3 审批逻辑已经进入“上下文落人”阶段，不能再按静态角色表简单查人

从 `sys_role`、`audit_step_def`、`audit_step_instance` 可以看出：

- `ROLE_APPROVER`、`ROLE_LEADER`、`ROLE_FINAL_APPROVER` 是合并角色
- 同一个 `role_id` 在不同步骤里代表不同实际审批人
- “分管校领导”还是全局席位

因此审批路由必须至少依赖：

- `flow_id` / `entity_type`
- `step_name` / `step_def_id`
- `requester_org_id`
- `approver_org_id`
- 必要时还要区分全局席位

### 3.4 `workflow_task` 不能继续当审批主事实表

它适合做：

- 待办箱
- 已办箱
- 我发起
- 催办视图

但不适合做：

- 审批状态唯一事实来源
- 审批节点推进依据
- 历史追溯唯一来源

审批事实应该以 `audit_instance` + `audit_step_instance` 为准。

### 3.5 当前最大的主链路缺口是“填报有数据，审批没闭环”

种子里已经有：

- `plan_report`
- `plan_report_indicator`
- `attachment`
- `alert_event`

但审批只完整建到了 `PLAN`。

这意味着当前系统更像：

1. 计划包可以下发审批
2. 月报可以填
3. 但月报审批链还没有标准化落在同一套工作流里

这个缺口不补，业务上就会长期出现：

- 填报状态和审批状态分裂
- 报表数据无法确定是否为正式口径
- 预警触发时无法区分“草稿偏差”还是“已提交偏差”

## 4. 目标业务架构

建议把系统重构为 6 个稳定业务上下文。

### 4.1 `sism-strategy`：计划与指标主链路

职责：

- 管理 `cycle`
- 管理 `plan`
- 管理 `indicator`
- 管理 `indicator_milestone`
- 负责计划包组装、指标树校验、计划下发前校验
- 负责计划生效后的指标联动

不负责：

- 月度填报事实
- 审批实例运行
- 通用待办箱

一句话定义：

`sism-strategy` 负责“今年给谁下发什么目标包，以及这个目标包由哪些指标构成”。

### 4.2 `sism-task`：任务中心

职责：

- 管理 `sys_task`
- 维护任务基础属性
- 管理任务分类与任务视图
- 承接未来非战略任务、专项任务、临时任务的扩展

不负责：

- `plan` 审批
- `plan_report` 填报
- `indicator` 归属与权重规则

一句话定义：

`sism-task` 负责“任务是什么”，不负责“任务包怎么审批”，也不负责“本月执行填了什么”。

### 4.3 `sism-execution`：填报与执行中心

职责：

- 管理 `plan_report`
- 管理 `plan_report_indicator`
- 管理填报附件绑定
- 提供月报草稿、提交、撤回、重填、查看历史能力
- 负责生成执行期快照和统计输入

不负责：

- 年度计划包下发
- 审批模板定义
- 审批待办投影

一句话定义：

`sism-execution` 负责“每个月实际执行了什么、填了什么、提交了什么证据”。

### 4.4 `sism-workflow`：审批中心

职责：

- 管理 `audit_flow_def`
- 管理 `audit_step_def`
- 管理 `audit_instance`
- 管理 `audit_step_instance`
- 统一承接 `PLAN`、`PLAN_REPORT` 等业务实体审批
- 输出待办、已办、我发起、审批历史

不负责：

- 维护年度计划内容
- 维护填报明细内容
- 做业务指标计算

一句话定义：

`sism-workflow` 只负责“谁审批、当前到哪一步、审批结果是什么”。

### 4.5 `sism-alert`：预警中心

职责：

- 管理 `alert_rule`
- 管理 `alert_event`
- 基于填报事实与目标里程碑计算偏差
- 提供提醒、关闭、处理记录

一句话定义：

`sism-alert` 负责“填报之后哪些指标偏了、偏了多少、有没有处理”。

### 4.6 `sism-iam` + `sism-organization`：人岗组织中心

职责：

- 管理 `sys_user`、`sys_role`、`sys_permission`
- 管理 `sys_org`
- 支撑审批路由、数据权限、组织视角查询

一句话定义：

它们不直接承接业务流程，但它们决定“谁能看、谁能填、谁能批”。

## 5. 重新规划后的主业务逻辑

建议以后端视角明确拆成 4 条标准主链路。

### 5.1 链路 A：年度计划包下发

适用场景：

- 战略发展部给职能部门下发年度计划包和任务
- 职能部门把本部门任务下的指标拆分并下达到二级学院

业务步骤：

1. `sism-strategy` 创建或维护 `plan`
2. `sism-task` 仅在职能部门层挂接一个或多个 `sys_task`
3. `sism-strategy` 维护 `indicator` 与 `indicator_milestone`
4. 如需学院承接，由职能部门将本部门任务下的指标拆分到学院
5. 学院承接后只拥有指标执行责任，不新增学院 `sys_task`
6. 发起下发时，由 `sism-strategy` 做前置校验
7. 校验通过后调用 `sism-workflow` 启动 `PLAN` 审批实例
8. `sism-workflow` 推进 `audit_instance` / `audit_step_instance`
9. 审批通过后回调 `sism-strategy`
10. `sism-strategy` 将 `plan.status` 更新为正式生效状态，并同步指标可执行状态

关键原则：

- 计划包审批属于 `sism-strategy` 发起、`sism-workflow` 执行
- `sism-task` 只提供职能部门层任务数据，不主导审批
- 学院承接链路以指标为中心，不新增学院任务

### 5.2 链路 B：月度填报

适用场景：

- 职能部门按月填报本部门任务与指标执行进展
- 二级学院按月填报其承接指标的执行进展

业务步骤：

1. `sism-execution` 根据 `plan + report_month` 获取或创建 `plan_report`
2. `sism-execution` 加载该计划包下所有可填报指标
3. 用户填写 `plan_report_indicator.progress`、说明、附件
4. 保存草稿时只更新填报事实，不进入审批
5. 提交时对填报完整性做校验

关键原则：

- 月报是“执行事实”，不是审批节点副本
- 填报应围绕 `plan_report` 聚合，而不是直接改 `indicator`

### 5.3 链路 C：月度填报审批

这是本次规划里最重要的新增标准化链路。

建议新增统一业务语义：

- `entity_type = PLAN_REPORT`

建议做法：

1. `sism-execution` 提交 `plan_report`
2. 调用 `sism-workflow` 启动 `PLAN_REPORT` 审批实例
3. `sism-workflow` 根据组织类型选用对应流程模板
4. 审批通过后回调 `sism-execution`
5. `sism-execution` 将 `plan_report.status` 置为 `APPROVED`
6. 同时把 approved 口径写入指标执行快照或统计投影

这样处理后，系统会形成清晰分层：

- `plan_report.status = DRAFT`
  - 只是草稿
- `plan_report.status = SUBMITTED` 或 `IN_REVIEW`
  - 已进审批
- `plan_report.status = APPROVED`
  - 可作为正式统计口径
- `plan_report.status = REJECTED`
  - 退回重新填报

关键原则：

- 填报审批必须复用 `sism-workflow`
- 不要在 `sism-execution` 内再做一套“轻审批”

### 5.4 链路 D：预警闭环

业务步骤：

1. `sism-alert` 基于 `plan_report_indicator.progress` 与 `indicator_milestone.target_progress` 计算偏差
2. 命中规则后创建 `alert_event`
3. 业务人员处理预警并记录处理说明
4. 看板按“未处理预警、已处理预警、严重级别”展示

关键原则：

- 预警的数据源优先采用“审批通过后的月报口径”
- 如需要支持草稿预警，应明确标注为“预测预警”，不能混用正式预警

## 6. 统一状态模型

### 6.1 `plan.status`

建议继续作为包级权威状态。

推荐状态集合：

- `DRAFT`
- `PENDING`
- `IN_REVIEW`
- `DISTRIBUTED`
- `REJECTED`
- `CLOSED`

说明：

- 当前种子里 `DRAFT`、`PENDING`、`DISTRIBUTED` 已经存在
- 后续代码中应避免由 `task.status` 或 `indicator.status` 反向主导 `plan.status`

### 6.2 `indicator.status`

建议降级为指标执行投影状态。

推荐语义：

- 仅表示指标自身是否可执行、是否已下发、是否停用
- 不再承担整个计划包审批状态

### 6.3 `plan_report.status`

建议明确为填报主状态。

推荐状态集合：

- `DRAFT`
- `SUBMITTED`
- `IN_REVIEW`
- `APPROVED`
- `REJECTED`

### 6.4 审批实例状态

- `audit_instance.status`
  - `IN_REVIEW`
  - `APPROVED`
  - `REJECTED`
  - `CANCELLED`

- `audit_step_instance.status`
  - `PENDING`
  - `APPROVED`
  - `REJECTED`

建议逐步收敛掉长期使用 `WAITING` 的做法，后续节点应在真正激活时再生成或再置为 `PENDING`。

## 7. 审批人路由规则

这是本次规划中必须落地的技术规则。

### 7.1 路由输入

审批人解析不能只传 `role_id`，至少要传：

- `entity_type`
- `flow_def_id`
- `step_def_id`
- `requester_id`
- `requester_org_id`
- `target_org_id`
- `approver_org_id`

### 7.2 路由原则

建议统一路由规则如下：

1. 提交节点
   - 不查审批人
   - 由发起人自动完成
2. 通用审批人节点
   - `role_id = ROLE_APPROVER`
   - 在当前审批组织内找对应审批席位
3. 通用领导节点
   - `role_id = ROLE_LEADER`
   - 根据步骤名区分“本组织领导”还是“全局分管校领导”
4. 通用终审节点
   - `role_id = ROLE_FINAL_APPROVER`
   - 根据步骤定义和组织上下文落到具体终审人

### 7.3 路由落表原则

审批路由结果只落运行态：

- `audit_step_instance.approver_id`
- `audit_step_instance.approver_org_id`

不要把“角色规则”复制进实例表做二次存储。

## 8. 建议的模块接口边界

### 8.1 `sism-strategy`

建议提供：

- 创建/更新计划包
- 维护计划包任务清单
- 维护指标树与里程碑
- 发起计划包审批
- 查询计划包详情聚合

### 8.2 `sism-task`

建议提供：

- 任务基础 CRUD
- 任务列表/筛选/搜索
- 按计划包查询任务
- 按组织查询任务

### 8.3 `sism-execution`

建议提供：

- 获取当月填报草稿
- 保存填报草稿
- 提交月报
- 撤回被驳回月报
- 查询历史月报
- 查询月报附件

### 8.4 `sism-workflow`

建议提供：

- 启动审批实例
- 批准
- 驳回
- 撤回
- 查询待办
- 查询已办
- 查询我发起
- 查询审批轨迹

## 9. 需要补齐的数据与模型

基于当前种子，建议优先补 4 个缺口。

### 9.1 为 `plan_report` 增加正式审批模板

建议新增：

- `audit_flow_def` for `PLAN_REPORT`
- `audit_step_def` for `PLAN_REPORT`
- 必要的 `workflow_task` 投影逻辑

这是最优先事项，因为它直接决定填报是否真正闭环。

### 9.2 为审批增加标准业务回调

建议由 `sism-workflow` 只负责审批过程，然后通过业务回调通知：

- `sism-strategy`
- `sism-execution`

不要让 `workflow` 模块直接改大量业务表。

### 9.3 为月报建立“审批通过口径”快照

当前 `indicator.progress` 与 `plan_report_indicator.progress` 可能会同时存在。

建议后续明确：

- `plan_report_indicator.progress`
  - 月度事实
- 指标聚合进度
  - 通过独立统计口径或投影计算得出

不要让月报提交直接覆盖指标主值，除非已经审批通过并且规则明确。

### 9.4 为附件建立业务归属规范

当前附件模型已经比较合理。

建议继续保持：

- `attachment` 只管物理文件
- `plan_report_indicator_attachment` 管业务绑定

不要把附件字段再塞回 `plan_report_indicator` 主表。

## 11. 最终建议

结合你们现在的表结构，我建议后端以后统一按下面这句话执行：

`sism-strategy` 负责下发什么目标包，`sism-task` 负责任务是什么，`sism-execution` 负责本月填了什么，`sism-workflow` 负责谁来批，`sism-alert` 负责哪些执行结果偏了。

其中有一条边界现在已经可以定死：

- `sism-task` 的任务边界只到职能部门
- 学院侧没有独立任务，只有从职能部门任务拆分下来的指标承接

如果只抓一个最关键动作，那就是：

优先把 `plan_report` 补进正式审批主链路，并把“填报事实”和“审批结果”彻底拆清。

因为你们当前数据库已经把“计划包审批”做出来了，下一步真正决定系统是否能稳定支撑核心业务的，不是再加更多任务字段，而是把“月报填报审批闭环”补完整。

## 12. 当前种子需要同步修正的点

按你刚刚确认的真实业务口径，当前种子里至少有两类关系需要后续清理：

1. `sys_task-data.sql` 中学院任务行不应继续保留
   - 如 `41004`、`41021` 到 `41027`
2. `indicator-data.sql` 中学院指标不应再挂到学院任务下
   - 例如 `2007`、`2008` 当前挂在 `task_id = 41004`

也就是说，正确模型应是：

- 职能部门任务是主任务
- 学院指标是从职能部门任务继续拆解出来的执行指标
- 学院只作为 `indicator.target_org_id` 和 `plan_report.report_org_id` 的承接方存在
