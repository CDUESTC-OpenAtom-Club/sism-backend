# SISM 企业级优化方案 - 需求文档

## 概述

本文档定义了战略指标管理系统 (SISM) 的企业级优化需求，旨在提升系统的安全性、可靠性、可维护性和性能，使其满足多人协作的企业级应用标准。

## 审查发现的问题

### 🔴 高优先级问题

1. **安全配置硬编码**: JWT 密钥、数据库密码等敏感信息硬编码在代码中
2. **生产环境调试日志**: 前端 API 层包含大量 console.log 调试信息
3. **Token 存储不安全**: JWT Token 直接存储在 localStorage，易受 XSS 攻击
4. **缺少请求重试机制**: API 请求失败后没有自动重试逻辑
5. **内存泄漏风险**: 频率限制使用内存 Map 存储，无持久化和集群支持

### 🟡 中优先级问题

6. **缺少 API 版本控制**: 没有 API 版本管理机制
7. **错误处理不一致**: 前后端错误格式不统一
8. **缺少请求幂等性**: 敏感操作缺少幂等性保护
9. **日志级别不可配置**: 无法动态调整日志级别
10. **缺少性能监控**: 没有前端性能指标收集

### 🟢 低优先级问题

11. **代码重复**: 多处存在相似的数据映射逻辑
12. **类型定义分散**: TypeScript 类型定义可进一步优化
13. **测试覆盖不足**: 部分关键路径缺少测试

## 用户故事

### 第一轮优化：安全加固

#### Story 1.1: 敏感配置外部化
作为系统管理员，我希望所有敏感配置都从环境变量读取，以便在不修改代码的情况下安全部署到不同环境。

**验收标准:**
- 1.1.1 JWT 密钥必须从环境变量读取，禁止硬编码默认值
- 1.1.2 数据库连接信息必须从环境变量读取
- 1.1.3 前端 API 签名密钥必须从环境变量读取
- 1.1.4 启动时检测必要环境变量，缺失时给出明确错误提示
- 1.1.5 提供 .env.example 模板，包含所有必需变量说明

#### Story 1.2: Token 安全存储
作为用户，我希望我的登录凭证被安全存储，以防止 XSS 攻击窃取我的身份信息。

**验收标准:**
- 1.2.1 Access Token 存储在内存中，不持久化到 localStorage
- 1.2.2 实现 Refresh Token 机制，Refresh Token 使用 HttpOnly Cookie 存储
- 1.2.3 Token 刷新失败时自动跳转登录页
- 1.2.4 页面刷新后能通过 Refresh Token 恢复会话
- 1.2.5 登出时同时清除 Access Token 和 Refresh Token

#### Story 1.3: 生产环境日志清理
作为开发者，我希望生产环境不输出调试日志，以防止敏感信息泄露并提升性能。

**验收标准:**
- 1.3.1 创建统一的日志工具类，支持日志级别控制
- 1.3.2 生产环境自动禁用 debug 和 info 级别日志
- 1.3.3 保留 warn 和 error 级别日志用于问题排查
- 1.3.4 敏感数据（Token、密码）在任何环境都不输出到日志
- 1.3.5 日志级别可通过环境变量动态配置

### 第二轮优化：可靠性增强

#### Story 2.1: API 请求重试机制
作为用户，我希望网络波动时系统能自动重试请求，以获得更稳定的使用体验。

**验收标准:**
- 2.1.1 GET 请求在网络错误或 5xx 错误时自动重试，最多 3 次
- 2.1.2 重试间隔采用指数退避策略（1s, 2s, 4s）
- 2.1.3 POST/PUT/DELETE 等非幂等请求默认不重试
- 2.1.4 可通过配置为特定接口启用/禁用重试
- 2.1.5 重试过程中显示加载状态，重试失败后显示友好错误提示

#### Story 2.2: 请求幂等性保护
作为用户，我希望重复提交表单时不会产生重复数据，以避免误操作造成的问题。

**验收标准:**
- 2.2.1 前端为每个写操作生成唯一请求 ID
- 2.2.2 后端在 5 分钟内拒绝相同请求 ID 的重复请求
- 2.2.3 幂等性检查覆盖指标创建、审批提交、进度填报等关键操作
- 2.2.4 重复请求返回原请求的结果而非错误
- 2.2.5 幂等性记录支持 Redis 存储（可选，默认内存）

#### Story 2.3: 分布式频率限制
作为系统管理员，我希望频率限制在集群环境下正常工作，以保护系统免受恶意请求。

**验收标准:**
- 2.3.1 频率限制支持 Redis 作为存储后端
- 2.3.2 单机模式下使用内存存储，无需额外配置
- 2.3.3 不同接口可配置不同的限制策略
- 2.3.4 登录接口限制更严格（每分钟 5 次）
- 2.3.5 触发限制时返回 429 状态码和重试时间

### 第三轮优化：可维护性提升

#### Story 3.1: 统一错误处理
作为开发者，我希望前后端有统一的错误格式，以便快速定位和处理问题。

**验收标准:**
- 3.1.1 定义统一的错误响应格式：{ code, message, details, requestId, timestamp }
- 3.1.2 前端错误拦截器统一转换为标准格式
- 3.1.3 后端异常处理器统一转换为标准格式
- 3.1.4 业务错误码文档化，包含错误码、描述、解决方案
- 3.1.5 错误响应包含 requestId 用于日志关联

#### Story 3.2: API 版本管理
作为开发者，我希望 API 支持版本控制，以便在不影响现有客户端的情况下迭代接口。

**验收标准:**
- 3.2.1 API 路径包含版本号：/api/v1/...
- 3.2.2 支持通过 Header 指定 API 版本（可选）
- 3.2.3 旧版本 API 标记为 deprecated 但继续可用
- 3.2.4 API 文档显示各版本差异
- 3.2.5 前端 API 客户端支持配置目标版本

#### Story 3.3: 代码质量工具集成
作为开发者，我希望有自动化的代码质量检查，以保持代码库的一致性和质量。

**验收标准:**
- 3.3.1 配置 ESLint 规则，禁止 console.log（warn/error 除外）
- 3.3.2 配置 Prettier 统一代码格式
- 3.3.3 配置 Husky pre-commit 钩子运行 lint 和类型检查
- 3.3.4 配置 commitlint 规范提交信息格式
- 3.3.5 CI 流程包含代码质量检查步骤

---

### 第四轮优化：性能优化

#### Story 4.1: 前端性能监控
作为产品经理，我希望能监控前端性能指标，以便发现和优化性能瓶颈。

**验收标准:**
- 4.1.1 收集 Core Web Vitals 指标（LCP, FID, CLS）
- 4.1.2 收集 API 请求耗时统计
- 4.1.3 收集页面加载时间和资源加载时间
- 4.1.4 性能数据可上报到后端或第三方服务
- 4.1.5 开发环境可在控制台查看性能报告

#### Story 4.2: API 响应缓存
作为用户，我希望常用数据能快速加载，以获得流畅的使用体验。

**验收标准:**
- 4.2.1 组织机构树数据缓存 5 分钟
- 4.2.2 指标列表支持条件缓存（数据未变化时使用缓存）
- 4.2.3 缓存支持手动刷新
- 4.2.4 缓存使用 ETag 或 Last-Modified 验证
- 4.2.5 缓存策略可通过配置调整

#### Story 4.3: 数据库查询优化
作为系统管理员，我希望数据库查询高效执行，以支持更多并发用户。

**验收标准:**
- 4.3.1 为常用查询字段添加索引
- 4.3.2 N+1 查询问题修复
- 4.3.3 大数据量查询使用分页
- 4.3.4 复杂查询添加执行计划分析
- 4.3.5 慢查询日志记录和告警

## 非功能性需求

### 安全要求
- 所有敏感数据传输必须使用 HTTPS
- 密码存储必须使用 BCrypt 加密
- JWT 密钥长度至少 256 位
- 会话超时时间不超过 24 小时

### 性能要求
- API 响应时间 P95 < 500ms
- 页面首次加载时间 < 3s
- 支持 100 并发用户

### 可用性要求
- 系统可用性 > 99.5%
- 计划内维护提前 24 小时通知
- 故障恢复时间 < 30 分钟

### 兼容性要求
- 支持 Chrome、Firefox、Edge 最新两个主版本
- 支持 1920x1080 及以上分辨率
- 支持移动端基本浏览（响应式布局）
